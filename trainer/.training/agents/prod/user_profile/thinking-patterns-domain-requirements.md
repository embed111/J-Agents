# 用户思维模式 - 需求分析场景

- 最后更新: 2026-02-24

## 场景特征
1. 关注点: 不满足于“表面需求”，强调挖掘底层动机与根本问题。
2. 交付偏好: 需求结论要可读、清晰，并可被团队直接使用。
3. 过程偏好: 先建立方法论，再落到模板和执行细节。
4. 能力定位（用户明示）: 自身并非专业需求分析人员，需要 agent 提供主动引导而非被动应答。
5. 新增痛点（用户明示）:
   - 过长上下文会降低处理质量（人和 agent 都受影响）。
   - 留痕文件过多会抬高每次启动的读取成本。

## 决策偏好（当前）
1. 偏好“可维护”的需求管理，而非一次性文档。
   - 置信度: 高
2. 偏好“跨层次”的需求表达（概述层 + 详情层）。
   - 置信度: 高
3. 可能重视“角色自约束”来保证长期质量。
   - 置信度: 中（推断）
4. 偏好“个性化引导”而非“通用默认分析流程”。
   - 置信度: 高

## 对应执行策略
1. 先明确目标、范围、验收标准，再补充功能细节。
2. 每次输出包含“已确认事实 / 推断假设 / 待确认问题”三类信息。
3. 以版本化方式维护需求文档，保留变更理由。
4. 默认先做“表达习惯对齐”再进入正式需求拆解，减少分析偏差。
5. 留痕策略采用“分层存储 + 启动最小读取”：
   - 启动仅读摘要索引与会话快照。
   - 详细记录按需回溯，不在每次启动全量加载。

## 待验证问题
1. 在需求冲突时，你更偏好“快速折中”还是“延后决策并补充调研”？
2. 你是否希望每个需求文档都附带优先级模型（如 MoSCoW）？
