# 需求概述

## 1. 项目背景与问题定义
1. 当前训练体系已能执行单轮训练，但“实际使用数据 -> 分析判定 -> 训练回写”的闭环仍依赖人工串联。
2. 用户希望减少心智负担，采用无感回流方式，让系统自动发现可训练样本并提供一键动作。
3. 本项目定位为独立产品：`workflow`，不是训练师方案内部章节。

## 2. 目标用户与核心场景
1. 目标用户：
   1. workflow 使用者（通过页面与 agent 对话完成日常工作）。
   2. 需求分析师（负责是否入训判定）。
   3. 训练师操作者（负责执行训练与回写）。
   4. workflow 运维者（负责在线升级切换与恢复）。
2. 核心场景：
   1. 在 workflow 页面直接与目标 agent 对话（统一入口 + 自动留痕）。
   2. 自动看到各 agent 新增对话。
   3. 自动形成待分析任务。
   4. 一键判定是否入训。
   5. 一键执行训练并回写结果。
   6. workflow 在线升级时自动切换，保持页面可用与可追溯。
   7. 新建会话时加载并冻结 `AGENTS.md` 上下文，保障职责一致。
   8. 并行会话运行时可明确识别“哪个会话正在运行”，并按需下钻链路。
   9. 用户通过 `$root` 引用全局根路径，减少跨工作区路径输入成本。

## 3. 业务目标与成功指标（可量化）
1. 自动采集时效：新增会话 60 秒内进入监控。
2. 自动建单覆盖率：新增会话自动生成分析任务覆盖率 >= 99%。
3. 判定回写时效：一键判定后 10 秒内可见决策结果。
4. 训练回写可追溯率：训练任务 100% 可追溯到 `task_id` 与报告文件。
5. 入口覆盖率：通过 workflow 入口发起的会话占比 = 100%（Phase 0 目标）。
6. 可用性目标：workflow 在线升级切换 `RTO <= 30 秒`，`RPO <= 5 秒`。
7. 对话体验目标：同模型同网络条件下，相比 CLI 基线，workflow 首 token 延迟增量 `p95 <= 200ms`。
8. 上下文一致性目标：100% 新建会话写入 `AGENTS.md` 快照元数据（`path/hash/loaded_at/version`）。
9. 运行可见性目标：并行会话运行状态刷新间隔 `p95 <= 1 秒`。

## 4. 范围界定（In Scope / Out of Scope）
1. In Scope：
   1. 统一入口网关（agent 请求必须经 workflow）。
   2. workflow 对话工作台（页面内直接与 agent 对话）。
   3. 会话采集与事件入库。
   4. 分析队列、训练队列状态管理。
   5. 分析师一键“是否入训”。
   6. 训练师一键执行与结果回写。
   7. 单机 A/B 运行槽位自动切换机制（用于在线升级）。
   8. 最小可视化看板（5 卡片 + 2 按钮）。
   9. 会话级 `AGENTS.md` 快照冻结（仅新会话生效）。
   10. 会话级运行状态可见与默认折叠链路下钻。
   11. 全局 `$root` 路径别名解析与越界拦截。
2. Out of Scope：
   1. 多工作流聚合大盘。
   2. 跨角色全自动编排。
   3. 复杂权限系统与多租户。
   4. 微服务拆分与分布式治理。
   5. 双机/跨机房容灾高可用架构。

## 5. 优先级与阶段规划
1. Phase 0（先活下来）：
   1. workflow 页面内 agent 对话工作台（最小可用）。
   2. 自动采集。
   3. 自动建队列。
   4. 一键判定入训。
   5. 一键训练回写。
   6. 单机 A/B 在线升级自动切换（MVP）。
2. Phase 1（稳定性增强）：
   1. 告警与重试策略完善。
   2. 在线升级演练自动化与切换策略调优（阈值、抖动控制）。
   3. 指标历史趋势与失败聚类。
3. Phase 2（扩展能力）：
   1. 多角色工作流接入。
   2. 更多编排能力与插件接口。

## 6. 风险、依赖与待确认事项
1. 风险：
   1. 事件采集不全导致漏训。
   2. 误判入训导致训练噪声。
   3. 过快扩展导致系统复杂度失控。
   4. agent 绕过 workflow 直连导致日志缺口。
   5. 在线升级切换失败导致页面短时不可用。
   6. 切换期间会话上下文丢失影响连续工作。
   7. `agent_search_root` 配置错误导致无可用 agent，会话创建被整体阻断。
   8. `$root` 路径别名若无强校验，存在越界访问风险。
2. 依赖：
   1. 现有 `trainer/` 目录和训练执行流程可调用。
   2. agent 日志输出具备最小结构化字段。
   3. 各 agent 目录下 `AGENTS.md` 具备可解析、可哈希的稳定文本格式。
3. 待确认事项：
   1. 第 4、5 张卡片最终命名是否固定为“最近判定结果/最近训练结果”。
   2. 采集周期默认值（建议 30~60 秒）。
   3. 强约束启用后的紧急旁路策略（谁可用、何时可用、如何补录）。
   4. CLI 丝滑体验的快捷键与交互映射清单（如中断、重试、历史检索）。

## 11. 本轮增量需求（2026-03-05）- 角色画像发布格式与预发布判定
1. 问题定义（用户明示）：
   1. 训练中心选择工牌后，需要先展示角色能力与角色知识等“角色画像”信息。
   2. 发布版本必须满足统一格式；不满足格式的记录不应被视为发布版本。
   3. 只要本地存在未被 `.gitignore` 忽略的改动（无论是否提交），都应视为预发布版本。
   4. 角色信息区后续需要“设置头像”能力；本轮仅要求可设置与默认头像，不设发布门禁。
2. 关键规则（用户确认）：
   1. 发布格式最小字段：`角色能力摘要`、`角色知识范围`、`技能（展示语义为 Agent Skills）`、`适用场景`、`版本说明`。
   2. 不符合发布格式的记录一律按普通提交处理，不进入发布版本能力范围。
   3. 预发布判定命令固定为 `git status --porcelain --untracked-files=normal`，非空即预发布。
3. 本轮交付索引：
   1. 需求详情：`docs/workflow/requirements/需求详情-角色画像发布格式与预发布判定.md`
   2. 详细设计：`docs/workflow/design/详细设计-角色画像发布格式与预发布判定.md`

## 12. 本轮增量需求（2026-03-06）- 角色画像体验优化（上传头像 + 简历式展示）
1. 用户反馈（明示）：
   1. 对当前角色画像效果不满意，尤其是头像设置方式与能力/知识技能展示方式。
2. 本轮确认（用户明示）：
   1. 头像设置方式：采用本地文件上传（`A`）。
   2. 不满意维度：字段内容、信息结构、展示样式均需调整（`D`）。
   3. 展示目标：整体效果接近“简历”，应做到“看一眼就知道该 agent 能做什么”。
   4. 头像上传限制：`png/jpg/webp` 且 `<=2MB`（已确认）。
   5. 展示形态：采用 `A`（单栏阅读流）。
   6. 头像区进一步收敛为“只展示一个头像 + 一个更换头像入口”，保持界面整洁。
   7. 技能区展示内容应为 `agent skill`，而不是抽象化技能标签。
   8. 当前阶段 `Agent Skills` 的展示口径应贴近当前 agent 工作区 `.codex/skills/` 目录下真实存在的技能项。
3. 本轮收敛口径：
   1. 在原“角色画像发布格式与预发布判定”主题内升级，不新开独立主题。
   2. 角色画像首屏改为“简历式信息架构”，`我能做什么` 作为首要信息块。
   3. 头像能力由“可设置入口”收敛为“单头像展示 + 更换头像入口 + 默认头像回退”。
   4. 技能区展示语义固定为 `Agent Skills`，要求直接呈现 agent skill 清单。
   5. `Agent Skills` 默认映射当前工作区 `.codex/skills/` 本地技能，当前阶段不混入系统级全局 skills。
   6. 详情主体区固定单栏自上而下，不采用双栏并列主体区。
4. 本轮交付索引（更新）：
   1. 需求详情（更新版）：`docs/workflow/requirements/需求详情-角色画像发布格式与预发布判定.md`
   2. 详细设计（更新版）：`docs/workflow/design/详细设计-角色画像发布格式与预发布判定.md`
