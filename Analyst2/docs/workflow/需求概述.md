# 需求概述

## 1. 项目背景与问题定义
1. 当前训练体系已能执行单轮训练，但“实际使用数据 -> 分析判定 -> 训练回写”的闭环仍依赖人工串联。
2. 用户希望减少心智负担，采用无感回流方式，让系统自动发现可训练样本并提供一键动作。
3. 本项目定位为独立产品：`workflow`，不是训练师方案内部章节。

## 2. 目标用户与核心场景
1. 目标用户：
   1. workflow 使用者（通过页面与 agent 对话完成日常工作）。
   2. 需求分析师（负责是否入训判定）。
   3. 训练师操作者（负责执行训练与回写）。
   4. workflow 运维者（负责在线升级切换与恢复）。
2. 核心场景：
   1. 在 workflow 页面直接与目标 agent 对话（统一入口 + 自动留痕）。
   2. 自动看到各 agent 新增对话。
   3. 自动形成待分析任务。
   4. 一键判定是否入训。
   5. 一键执行训练并回写结果。
   6. workflow 在线升级时自动切换，保持页面可用与可追溯。
   7. 新建会话时加载并冻结 `AGENTS.md` 上下文，保障职责一致。
   8. 并行会话运行时可明确识别“哪个会话正在运行”，并按需下钻链路。
   9. 用户通过 `$root` 引用全局根路径，减少跨工作区路径输入成本。

## 3. 业务目标与成功指标（可量化）
1. 自动采集时效：新增会话 60 秒内进入监控。
2. 自动建单覆盖率：新增会话自动生成分析任务覆盖率 >= 99%。
3. 判定回写时效：一键判定后 10 秒内可见决策结果。
4. 训练回写可追溯率：训练任务 100% 可追溯到 `task_id` 与报告文件。
5. 入口覆盖率：通过 workflow 入口发起的会话占比 = 100%（Phase 0 目标）。
6. 可用性目标：workflow 在线升级切换 `RTO <= 30 秒`，`RPO <= 5 秒`。
7. 对话体验目标：同模型同网络条件下，相比 CLI 基线，workflow 首 token 延迟增量 `p95 <= 200ms`。
8. 上下文一致性目标：100% 新建会话写入 `AGENTS.md` 快照元数据（`path/hash/loaded_at/version`）。
9. 运行可见性目标：并行会话运行状态刷新间隔 `p95 <= 1 秒`。

## 4. 范围界定（In Scope / Out of Scope）
1. In Scope：
   1. 统一入口网关（agent 请求必须经 workflow）。
   2. workflow 对话工作台（页面内直接与 agent 对话）。
   3. 会话采集与事件入库。
   4. 分析队列、训练队列状态管理。
   5. 分析师一键“是否入训”。
   6. 训练师一键执行与结果回写。
   7. 单机 A/B 运行槽位自动切换机制（用于在线升级）。
   8. 最小可视化看板（5 卡片 + 2 按钮）。
   9. 会话级 `AGENTS.md` 快照冻结（仅新会话生效）。
   10. 会话级运行状态可见与默认折叠链路下钻。
   11. 全局 `$root` 路径别名解析与越界拦截。
2. Out of Scope：
   1. 多工作流聚合大盘。
   2. 跨角色全自动编排。
   3. 复杂权限系统与多租户。
   4. 微服务拆分与分布式治理。
   5. 双机/跨机房容灾高可用架构。

## 5. 优先级与阶段规划
1. Phase 0（先活下来）：
   1. workflow 页面内 agent 对话工作台（最小可用）。
   2. 自动采集。
   3. 自动建队列。
   4. 一键判定入训。
   5. 一键训练回写。
   6. 单机 A/B 在线升级自动切换（MVP）。
2. Phase 1（稳定性增强）：
   1. 告警与重试策略完善。
   2. 在线升级演练自动化与切换策略调优（阈值、抖动控制）。
   3. 指标历史趋势与失败聚类。
3. Phase 2（扩展能力）：
   1. 多角色工作流接入。
   2. 更多编排能力与插件接口。

## 6. 风险、依赖与待确认事项
1. 风险：
   1. 事件采集不全导致漏训。
   2. 误判入训导致训练噪声。
   3. 过快扩展导致系统复杂度失控。
   4. agent 绕过 workflow 直连导致日志缺口。
   5. 在线升级切换失败导致页面短时不可用。
   6. 切换期间会话上下文丢失影响连续工作。
   7. `agent_search_root` 配置错误导致无可用 agent，会话创建被整体阻断。
   8. `$root` 路径别名若无强校验，存在越界访问风险。
2. 依赖：
   1. 现有 `trainer/` 目录和训练执行流程可调用。
   2. agent 日志输出具备最小结构化字段。
   3. 各 agent 目录下 `AGENTS.md` 具备可解析、可哈希的稳定文本格式。
3. 待确认事项：
   1. 第 4、5 张卡片最终命名是否固定为“最近判定结果/最近训练结果”。
   2. 采集周期默认值（建议 30~60 秒）。
   3. 强约束启用后的紧急旁路策略（谁可用、何时可用、如何补录）。
   4. CLI 丝滑体验的快捷键与交互映射清单（如中断、重试、历史检索）。

## 10. 本轮增量需求（2026-02-26）
1. 训练输入源统一：
   1. “工作记录”卡直接使用会话对话内容作为训练输入源，不再依赖独立中间记录来源。
2. 消息分析状态：
   1. 每条对话消息都具备分析状态：`未分析` / `已分析`。
   2. “已分析”不等于“已生成训练计划”；允许“已分析但无训练价值”。
3. 选择门禁：
   1. 每次仅允许勾选“包含至少一条未分析消息”的会话进入分析。
4. 分析上下文规则：
   1. 每次分析必须基于该会话“完整对话上下文”，不允许仅截取局部片段直接分析。
5. 分析与训练计划映射：
   1. 分析结果需与训练计划建立映射关系（`analysis_run -> 0..N training_plan_items`）。
   2. 对无训练价值内容需保留“无价值原因”标记（例如：`no_training_value`）。
6. 失败与回滚策略：
   1. 分析失败默认回滚为“未分析”。
   2. 若可判定“上文缺失”，允许直接标记“已分析 + 无价值 + 上文缺失”。
   3. 若为“下文缺失”，保持“未分析”，等待新增对话后再次分析。
7. 会话分析状态管理：
   1. 支持在“会话分析状态”视图删除单条聊天记录，并写入审计留痕。
   2. 若会话已分析且已生成训练计划（存在 `training_plan_items > 0`），则该会话内聊天记录禁止删除。
8. CRUD 基线规则（需求治理）：
   1. 每个新增/变更功能必须明确是否需要 `Create/Read/Update/Delete` 四类操作。
   2. 对禁用操作必须给出禁用原因、触发条件与错误码。
9. 关键风险补充：
   1. 删除能力与训练计划追溯天然冲突，需以“追溯完整性优先”作为默认策略。
10. 能力扩展评估（skill/agent）：
   1. 评估结论：建议新增本地 skill（暂定名：`crud-requirement-gate`）。
   2. 触发理由：用户明确要求“每个功能必须进行 CRUD 审查”，属于高频重复动作。
   3. 预期收益：降低漏项与口径漂移，提升跨工作区交付一致性。
   4. 采用成本：低到中（主要为模板与检查清单固化）。
   5. 采用决策：`采用`（待你确认后再创建 skill 文件）。

## 11. 本轮增量需求（2026-02-27）
1. 会话入口角色/目标来源统一：
   1. 不再手工填写“角色/目标”文本。
   2. 必须从所选 agent 工作区 `AGENTS.md` 自动提取“角色/目标/职责边界”。
2. 会话级强约束执行：
   1. 提取出的“角色/目标/职责边界”必须在会话创建后冻结为会话上下文。
   2. 后续每一轮对话都必须按该上下文执行，不允许漂移到通用助手口径。
3. 训练链路一致性：
   1. 分析与训练决策链路也必须受同一份会话级“角色/目标”约束，而非仅影响聊天回复。
4. 最小目标（用户确认）：
   1. 采用目标 `C`：角色/目标同时驱动“回答行为 + 训练决策”。
5. 关键风险补充：
   1. 若只做“展示卡片”而不进入 prompt/决策链路，会出现“看似有功能、实际无效果”的假闭环。
   2. 若 `AGENTS.md` 提取失败但继续放行，会导致职责漂移与结果不可追溯。
6. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`。
   2. 新增 skill：建议新增 `agent-context-chain-gate`（会话入口参数是否真正进入 prompt/事件/训练链路的门禁检查）。
   3. 采用成本：中（需补链路断言脚本与回归用例）。
7. 身份/职责类问答要求：
   1. 当用户询问“你是谁/你的职责是什么/你能做什么”时，必须基于会话冻结策略返回规范化自我说明。
   2. 回复至少包含“agent 身份、职责边界、可提供能力”，并与训练决策口径保持一致。
8. 角色/目标清晰度与兜底确认：
   1. 提取策略需计算清晰度评分（完整性、具体性、一致性、可追溯性）。
   2. 低置信度场景必须弹窗人工确认后方可继续，拒绝则阻断执行。
9. 训练补齐闭环：
   1. 对人工确认/编辑过的兜底策略，系统需自动生成“AGENTS 策略补齐训练任务”，逐步减少兜底依赖。

## 12. 本轮增量需求（2026-02-27）- 会话创建门禁与策略缓存
1. 会话入口默认态：
   1. 首次打开主界面时，“会话入口”默认不选中任何 agent。
   2. 在未选择 agent 前，“创建会话”按钮必须禁用并给出可读提示（例如：`请先选择 agent`）。
2. 角色策略分析门禁：
   1. 用户选择 agent 后，系统先执行“角色/目标/职责边界”分析流程。
   2. 分析未完成前（含 `analyzing`、`needs_confirm`、`failed`），必须保持“创建会话”禁用。
   3. 仅当策略状态为 `ready`（或人工确认通过）后，才允许创建会话。
3. 角色分析缓存与失效：
   1. 角色分析结果允许缓存，至少包含：`agent_path`、`agents_hash`、`agents_mtime`、`parse_status`、`clarity_score`、`cached_at`。
   2. 创建会话前必须比较缓存版本与 `AGENTS.md` 文件状态；若缓存早于文件修改时间或哈希不一致，则强制重算。
   3. 缓存命中时应复用结果以缩短等待时长，缓存失效时必须重新分析，不得静默使用旧值。
4. 手动输入角色/职责功能定位：
   1. 手动输入仅作为受控兜底能力，不作为主路径。
   2. 建议通过管理员开关控制（默认关闭）；仅在自动提取失败或低置信度时可见。
   3. 手动输入/编辑后必须写审计并触发“策略补齐训练任务”，推动回写到 `AGENTS.md` 标准章节。
5. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`（当前问题核心是入口门禁与缓存一致性，不是角色分工不足）。
   2. 新增 skill：建议 `agent-policy-cache-gate`（检查“缓存是否过期、门禁是否生效、会话是否被提前放行”）。
   3. 采用成本：中（需增加缓存元数据管理、状态机断言与门禁回归用例）。

## 13. 本轮增量需求（2026-02-28）- 角色策略详情信息顺序与评分机制优化
1. 问题定义（用户明示）：
   1. 当前页面不满意点主要是“内容摆放顺序与模块划分不符合操作预期”，不是“信息过多”。
   2. 用户进入页面后，第一时间应看到 `角色/目标` 与 `职责边界`，而不是 `门禁与来源`。
   3. 当前评分细节“还差点意思”，需补强评分机制的完备性与可解释性。
2. 目标：
   1. 将角色策略详情页改为“按决策顺序阅读”的信息架构。
   2. 保留治理与审计信息，但降级为默认折叠层，避免打断主决策路径。
   3. 升级门禁评分机制，支持“可理解、可追溯、可修复”。
3. 页面顺序（用户确认）：
   1. 第一屏：`角色/目标`。
   2. 第二层：`职责边界`（合并职责与限制，含必须/禁止/前置条件）。
   3. 第三层：`角色设定门禁得分项`。
   4. 第四层：`门禁与来源`（默认折叠，可展开）。
   5. 第五层：`AGENTS.md 原文证据`（默认折叠，可展开）。
   6. 第六层：`风险提示`。
   7. 编辑区顺序保持现状。
4. 默认折叠策略（用户更新确认）：
   1. 默认展开：`角色/目标`、`职责边界`、`风险提示`。
   2. 默认折叠：`角色设定门禁得分项`、`门禁与来源`、`AGENTS.md 原文证据`。
5. 评分机制优化方向：
   1. 评分维度从“结果分数”升级为“维度评分 + 扣分原因 + 证据映射 + 修复建议”。
   2. 每条扣分必须可追溯到证据片段，且给出最小修复动作。
6. 风险补充：
   1. 若继续把 `门禁与来源` 置于首屏，用户会先进入系统治理语义，影响任务判断效率。
   2. 若评分仍仅给总分、无扣分证据，人工确认质量难稳定，长期会形成“分数黑盒”。
7. 成功判定（本增量）：
   1. 首屏可在一次视线扫描内完成“是否继续会话”的判断。
   2. 评分卡支持从“为什么扣分”直接定位到“改哪段 AGENTS.md”。
8. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`（问题集中在信息架构与评分机制，不在角色分工）。
   2. 新增 skill：建议 `policy-score-explainability-gate`（校验评分维度完整性、扣分证据映射与修复建议可用性）。
   3. 采用成本：中（需要评分字段扩展、证据定位与回归样本维护）。
