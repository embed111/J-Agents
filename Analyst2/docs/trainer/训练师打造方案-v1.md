# 训练师打造方案 v1.1（可直接在新工作区执行）

## 1. 目标与范围
1. 目标：在不破坏当前可用分析师的前提下，训练出可晋升的新分析师。
2. 范围：先落地“分析师训练闭环”，后续再扩展到架构师、开发、测试。
3. 关键约束：
   1. 生产分析师不参与训练写入。
   2. 高风险优先级为“数据/安全风险 > 用户体验风险”。
   3. 候选体连续 3 轮无关键退化才可晋升。
   4. 训练强度按 token 预算控制，不用固定每日任务量。

## 2. 角色定义
1. `analyst_prod`：生产分析师，只读，对外交付。
2. `analyst_cand`：候选分析师，可训练，不直接覆盖生产。
3. `trainer_judge`：任务分配、评分、反馈、晋升建议。
4. `trainer_audit`：审计评分一致性，防止评判漂移。

## 5. 任务单与任务指派设计（放入 `trainer/queue/inbox/`）
```yaml
task_id: "REQ-20260224-001"
title: "示例任务"
source: "real_request"
created_at: "2026-02-24T12:00:00+08:00"
priority: "P0|P1|P2|P3"
risk_level: "high|medium|low"
risk_type: "data_security|user_experience|general"
target_role: "analyst"
dispatch_mode: "manual"
execution_route: "auto_by_risk"
acceptance:
  - "输出含项目背景、目标用户、范围边界、验收标准"
  - "高风险场景明确列出数据/安全风险"
notes: "补充上下文"
```

### 5.1 队列流转（MVP）
1. `inbox`：待分配任务。
2. `processing`：已分配、执行中。
3. `done`：完成并产出评测报告。
4. `blocked`：信息不足或规则冲突，等待补充。

### 5.2 指派规则（MVP）
1. 排序优先级：
   1. `priority`（`P0 > P1 > P2 > P3`）
   2. `risk_type`（`data_security > user_experience > general`）
   3. `created_at`（越早越先）
2. 路由规则：
   1. 高风险任务：`analyst_prod` 对外交付，`analyst_cand` 仅影子执行。
   2. 中低风险任务：允许候选体参与 canary（不直接覆盖生产）。
3. 关键约束：
   1. 信息不足时不得假设推进，直接转 `blocked` 并生成澄清问题。
   2. 所有分配决策必须写入 `trainer/runs/<task_id>-dispatch.md`。

### 5.3 你如何“给训练师指定任务”（可直接执行）
1. 在 `trainer/queue/inbox/` 新建一个任务 YAML 文件（文件名建议：`<task_id>.yaml`）。
2. 至少填写：`task_id`、`title`、`priority`、`risk_level`、`risk_type`、`acceptance`。
3. 若有上下文附件，放到 `trainer/queue/inbox/<task_id>/` 目录。
4. 训练师读取后会：
   1. 按规则挑选任务。
   2. 生成分配记录到 `trainer/runs/`。
   3. 执行并输出报告到 `trainer/reports/`。

### 5.4 任务模板（推荐最小字段）
```yaml
task_id: "REQ-YYYYMMDD-XXX"
title: "一句话说明任务目标"
source: "real_request"
created_at: "YYYY-MM-DDTHH:mm:ss+08:00"
priority: "P1"
risk_level: "medium"
risk_type: "general"
target_role: "analyst"
dispatch_mode: "manual"
execution_route: "auto_by_risk"
acceptance:
  - "验收标准1"
  - "验收标准2"
notes: "补充约束或上下文链接"
```

## 6. 单轮执行流程（严格按顺序）
1. `trainer_judge` 读取 `inbox` 首条任务，移到 `processing`。
2. 按 `risk-policy.yaml` 判定是否高风险。
3. 同题执行：
   1. `analyst_prod` 生成生产答案。
   2. `analyst_cand` 生成候选答案（高风险任务仅影子，不对外）。
4. `trainer_judge` 按 `score-rubric.yaml` 给出维度分、总分、关键退化判断。
5. `trainer_audit` 复核评分一致性并给出审计结论。
6. 若无关键退化：允许对候选体的 `mutable-files.txt` 中内容做小步改进。
7. 若有关键退化：冻结候选改动并回滚到上一个候选快照。
8. 生成报告 `trainer/reports/<task_id>-report.md`。
9. 任务归档到 `done` 或 `blocked`。

## 7. 晋升与回滚
1. 晋升条件：
   1. 连续 3 轮无关键退化。
   2. 高风险回归任务通过。
   3. 最近 3 轮候选总分均不低于生产体。
2. 晋升动作：
   1. 当前生产体备份到 `trainer/snapshots/prod-<timestamp>/`。
   2. 候选体复制覆盖生产体。
   3. 旧生产体作为新候选体初始版本。
3. 回滚动作：
   1. 发现关键退化立即回滚到最近生产快照。
   2. 输出回滚报告并记录触发原因。

## 8. 可用门槛（训练停止条件）
1. 高风险任务 0 失败。
2. 连续 3 轮无关键退化。
3. 候选体需求对齐得分连续 3 轮不低于生产体。
4. 满足以上条件后，进入“稳态维护模式”（仅增量训练）。

## 9. 新工作区执行清单（从零开始）
1. 建立目录结构与策略文件。
2. 复制当前可用分析师到 `agents/prod/`。
3. 复制 `agents/prod/` 到 `agents/candidates/c1/` 作为初始候选体。
4. 向 `trainer/queue/inbox/` 投放 3 条真实任务。
5. 连续执行单轮流程，直到满足可用门槛。
6. 完成首轮晋升后，再考虑引入第二候选体或扩展到其他角色。

## 10. 明确不做（防止系统失控）
1. 不直接改动生产分析师文件。
2. 不允许候选体改动 `immutable-files.txt` 内文件。
3. 不在高风险任务上让候选体直接对外交付。
4. 不在指标未定义时扩展到多角色全链路。

## 11. 后续扩展顺序（建议）
1. 先稳定分析师训练闭环。
2. 再训练架构师。
3. 再训练开发工程师。
4. 最后训练测试工程师并打通端到端项目流水线。

## 12. 对话式指派任务（替代手工写 YAML）
### 12.1 目标
1. 用户继续使用自然对话下达任务。
2. 训练师自动将对话转成结构化任务卡并入队。
3. 缺少关键信息时，训练师先追问再入队。

### 12.2 对话入口角色
1. 新增 `trainer_intake`（可由 `trainer_judge` 兼任）。
2. 职责：
   1. 解析用户输入为任务字段。
   2. 校验必填字段。
   3. 生成并落盘 `<task_id>.yaml` 到 `trainer/queue/inbox/`。

### 12.3 对话指派协议（MVP）
1. 用户发起（自然语言）：
   1. “给分析师一个任务：整理聊天系统需求，优先级 P1，重点看数据安全风险。”
2. `trainer_intake` 返回结构化复述：
   1. 目标角色：`analyst`
   2. 任务标题：`整理聊天系统需求`
   3. 优先级：`P1`
   4. 风险类型：`data_security`
   5. 验收标准：`待补充`
3. 若缺少关键字段（如验收标准、风险等级），先追问 1~3 个问题。
4. 用户确认后，自动落盘任务卡并回执：
   1. `task_id`
   2. 队列路径
   3. 路由结果（高风险/中低风险）

### 12.4 必填与可选字段
1. 必填：
   1. `target_role`
   2. `title`
   3. `priority`
   4. `risk_level`
   5. `risk_type`
   6. `acceptance`（至少 1 条）
2. 可选：
   1. `notes`
   2. `attachments`
   3. `deadline`

### 12.5 对话模板（你可直接用）
1. 简版：
   1. “指派任务给 `<角色>`：`<一句话目标>`，优先级 `<P1>`，风险 `<data_security>`，验收 `<...>`。”
2. 带约束版：
   1. “让 `<角色>` 处理 `<目标>`，限制 `<不做什么>`，重点风险 `<...>`，完成标准 `<...>`。”

## 13. 训练不同角色的方法（统一框架）
### 13.1 原则
1. 训练内核不变：队列、评分、晋升、回滚、审计共用。
2. 角色差异放在“角色包”中，不改训练主流程。

### 13.2 角色包目录

```text
    task-examples/
  architect/

    pass-gate.yaml
    task-examples/
  developer/
    role-policy.yaml
    score-rubric.yaml
    pass-gate.yaml
    task-examples/
  tester/
    role-policy.yaml
    score-rubric.yaml
    pass-gate.yaml
    task-examples/
```

### 13.3 角色切换逻辑
1. `target_role=analyst`：加载 `trainer/roles/analyst/*`。
2. `target_role=architect`：加载 `trainer/roles/architect/*`。
3. `target_role=developer`：加载 `trainer/roles/developer/*`。
4. `target_role=tester`：加载 `trainer/roles/tester/*`。

### 13.4 不同角色如何训练（MVP 顺序）
1. 分析师（入口角色）：
   1. 训练需求澄清、范围边界、验收定义。
2. 架构师：
   1. 训练模块划分、接口契约、风险隔离。
3. 开发：
   1. 训练按契约实现、回归修复、变更最小化。
4. 测试：
   1. 训练冒烟/SIT/UAT 用例设计与缺陷回归闭环。

### 13.5 跨角色训练闭环（简版）
1. 分析师产出需求基线。
2. 架构师消费需求基线并产出架构基线。
3. 开发消费架构基线并产出实现与变更说明。
4. 测试消费实现产出测试报告。
5. 培训师统一评分并决定是否回流上游角色重训。

## 14. 操作记录与复盘机制（用于分析成功/失败原因）
### 14.1 记录目标
1. 能回放“用户输入 -> 指派决策 -> 角色输出 -> 评分结论 -> 晋升/回滚”全链路。
2. 能定位失败根因，而不只看到结果失败。
3. 保持低成本读取：启动仅读摘要，详细日志按需展开。

### 14.2 日志目录
```text
trainer/logs/
  conversations/          # 对话原始记录（按 task_id 分文件）
  dispatch/               # 指派决策日志
  execution/              # 角色执行摘要与输出路径
  scoring/                # 评分与审计结果
  decisions/              # 晋升/回滚/冻结决策
  summaries/
    daily-summary.md      # 每日摘要（启动优先读）
    failure-cases.md      # 失败案例索引
```

### 14.3 统一事件格式（建议 JSONL）
```json
{
  "event_id": "evt-20260224-001",
  "timestamp": "2026-02-24T13:00:00+08:00",
  "task_id": "REQ-20260224-001",
  "stage": "intake|dispatch|execute|score|audit|decision",
  "actor": "trainer_intake|trainer_judge|trainer_audit|analyst_prod|analyst_cand",
  "target_role": "analyst",
  "risk_level": "high",
  "risk_type": "data_security",
  "input_summary": "用户对话摘要或任务摘要",
  "output_ref": "相对路径",
  "score": 8.4,
  "critical_regression": false,
  "decision": "keep_training|promote|rollback|blocked",
  "reason_tags": ["missing_acceptance", "risk_misjudge"],
  "notes": "补充说明"
}
```

### 14.4 成功/失败标签体系（最小可用）
1. 成功标签：
   1. `goal_hit`：命中任务目标。
   2. `risk_handled`：风险识别和处理符合策略。
   3. `clear_handoff`：可被下游直接消费。
2. 失败标签：
   1. `misunderstood_context`：语境理解偏差。
   2. `missing_acceptance`：验收标准缺失或不可测。
   3. `risk_misjudge`：风险等级判断错误。
   4. `assumption_unconfirmed`：未确认假设直接推进。
   5. `handoff_unusable`：交接内容不可执行。

### 14.5 记录触发点（必须写日志）
1. 对话指派确认时（`trainer_intake`）。
2. 风险路由决策时（`trainer_judge`）。
3. 角色输出落盘时（`analyst_prod` / `analyst_cand`）。
4. 评分与审计完成时（`trainer_judge` / `trainer_audit`）。
5. 晋升/回滚决策时（`trainer_judge`）。

### 14.6 复盘流程（每轮可执行）
1. 从 `decisions/` 取最近一条失败决策。
2. 回溯同 `task_id` 的 `conversations -> dispatch -> execution -> scoring`。
3. 提取前 3 个 `reason_tags` 作为根因。
4. 对应根因仅修改候选体可变文件，并记录“改动 -> 预期影响”。
5. 下一轮同类任务验证，判断是否复发。

### 14.7 启动读取优化（防止日志过重）
1. 启动仅读：
   1. `trainer/logs/summaries/daily-summary.md`
   2. `trainer/logs/summaries/failure-cases.md`
2. 详细日志按需读取，不做全量加载。
3. 每日结束自动生成摘要，归档原始日志。

## 15. 人为调控训练方向（Human-in-the-Loop）
### 15.1 目标
1. 保持“训练师 agent 自动评分”为默认。
2. 允许人类在不接管全流程的前提下，调控训练方向和策略权重。
3. 防止训练偏航（只会优化分数，不优化真实价值）。

### 15.2 控制原则
1. 默认自动：训练师继续自动派单、评分、晋升建议。
2. 人类调向：人类主要调“方向”，不必逐题打分。
3. 关键仲裁：涉及晋升到生产或高风险策略变更时，允许人类一票否决。

### 15.3 新增策略文件
1. `trainer/policy/human-steering.yaml`
2. 建议模板：
```yaml
mode: "direction_control"   # direction_control | direction_and_score

focus_topics:
  - "需求澄清质量"
  - "风险识别准确性"

weight_overrides:
  requirement_alignment: 0.45
  risk_identification: 0.30
  clarity_actionability: 0.15
  assumption_labeling: 0.10

forbidden_optimizations:
  - "为提分牺牲可执行性"
  - "未确认假设直接推进"

promotion_guardrails:
  require_human_approval: true
  freeze_promotion: false

exploration_control:
  allow_aggressive_exploration: false
  max_mutation_per_round: 1
```

### 15.4 人类可执行的 4 种调控动作
1. 方向调权：
   1. 调整评分维度权重（例如提高风险识别权重）。
2. 任务导向：
   1. 注入“专项训练任务”（例如连续 5 轮只练需求澄清）。
3. 晋升控制：
   1. 打开/关闭 `freeze_promotion`，暂停候选体晋升。
4. 风险闸门：
   1. 临时收紧高风险条件或降级候选外放权限。

### 15.5 决策合成规则（简版）
1. 自动评分仍是主信号。
2. 人类方向控制通过策略文件影响：
   1. 评分维度权重
   2. 任务采样分布
   3. 晋升/冻结门禁
3. 不建议每轮人工改分；建议每 3~5 轮做一次方向校准。

### 15.6 对话式调控指令（你可直接用）
1. “从现在开始，把风险识别权重提到 0.35，持续 10 轮。”
2. “接下来 5 轮只训练分析师的需求澄清，不做角色扩展。”
3. “冻结晋升，直到我手动解冻。”
4. “高风险任务继续只允许生产体对外交付。”

### 15.7 调控留痕（必须记录）
1. 每次人工调控都要落盘：
   1. `trainer/logs/decisions/human-steering-log.md`
2. 记录字段：
   1. 调控时间
   2. 调控人
   3. 调控项
   4. 预期影响
   5. 生效轮次
   6. 回退条件
3. 复盘时必须回答：
   1. 调控是否提升目标维度？
   2. 是否引入副作用（例如整体可执行性下降）？

## 16. 无感回流机制（生产直用 + 异步训练）
### 16.1 目标
1. 用户继续直接使用生产分析师，不增加额外交互步骤。
2. 每次工作自动留痕，不要求用户手工打标。
3. 周期性由需求分析师筛选训练样本，再交给训练师执行。

### 16.2 目录与文件
```text
trainer/feedback/
  incoming/
    events-YYYYMMDD.jsonl      # 自动回流事件
  triage/
    triage-YYYYMMDD.md         # 需求分析师筛选报告
  selected/
    selected-YYYYMMDD.yaml     # 入训任务清单
```

### 16.3 自动留痕事件格式（建议）
```json
{
  "event_id": "prod-20260224-001",
  "timestamp": "2026-02-24T15:00:00+08:00",
  "role": "analyst_prod",
  "scenario": "requirement_clarification",
  "input_summary": "用户问题摘要",
  "output_ref": "runs/prod-session-001.md",
  "risk_level": "high|medium|low",
  "risk_type": "data_security|user_experience|general",
  "result_hint": "success|partial|failure",
  "reason_tags": ["misunderstood_context", "missing_acceptance"],
  "notes": "可选"
}
```

### 16.4 周期筛选流程（需求分析师执行）
1. 每 24 小时读取 `trainer/feedback/incoming/` 新事件。
2. 生成 `triage-YYYYMMDD.md`，按以下规则筛选：
   1. 高风险失败样本：必选入训。
   2. 连续复发失败标签：必选入训。
   3. 高价值成功样本：抽样入训（防止只学失败）。
3. 输出 `selected-YYYYMMDD.yaml` 并写入 `trainer/queue/inbox/`。

### 16.5 触发训练的最小规则
1. 命中 `risk_level=high` 且 `result_hint!=success`。
2. 同一 `reason_tags` 在 48 小时内出现 >= 2 次。
3. 人工调控文件 `human-steering.yaml` 指定专项训练主题。

### 16.6 对用户可见体验
1. 你只需像平常一样使用分析师。
2. 无需每轮手工派单、打标、填模板。
3. 训练是否发生由系统在后台异步决策与执行。

## 17. 快速造角通道（训练初期优先）
### 17.1 目标
1. 快速创建“可用角色”，先满足业务启动。
2. 后续再进入常规训练闭环持续优化。

### 17.2 快速造角输入（对话即可）
1. 一句话角色目标。
2. 主要场景（1~3 个）。
3. 风险偏好（默认继承全局策略）。

### 17.3 快速造角输出
1. `agents/prod/<role>/AGENTS.md` 初稿。
2. `trainer/roles/<role>/role-policy.yaml` 初稿。
3. `trainer/roles/<role>/score-rubric.yaml` 初稿。
4. `trainer/roles/<role>/pass-gate.yaml` 初稿。

### 17.4 快速验收（最小 2 题）
1. 高风险任务 1 条（验证风险闸门）。
2. 常规任务 1 条（验证可执行性）。
3. 通过条件：
   1. 不触发关键退化。
   2. 输出可被下游消费。
   3. 评分不低于设定基线。

### 17.5 过渡到常规训练
1. 快速造角通过后标记 `status: usable_bootstrap`。
2. 自动纳入无感回流机制，按周期筛选继续训练。
