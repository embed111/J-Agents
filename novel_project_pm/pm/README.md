# PM治理目录

## 定位

本目录是 `novel_project_pm` 的项目治理、版本推进和日常节奏顶层真相源，结构模仿 `../workflow/pm`。

`novel/` 是唯一真实小说产出目录；`pm/` 只维护计划、状态、指标、门禁、审稿、运行手册和历史证据。

`pm/` 顶层只保留入口层，不承载具体版本现场。常规开工默认不全量阅读所有入口文件，先用轻量读链和自动化草案定位当前最高价值动作。

## 治理结构

| 路径 | 职责 |
|---|---|
| `README.md` | PM治理目录入口、目录分层和轻量读链 |
| `PM版本推进计划.md` | 稳定总规则、版本节奏、主线排序、反空转和高产高质计量规则 |
| `PM版本推进注意事项.md` | 稳定反省结论、后续切版注意事项和避免复发规则 |
| `PM版本目录导航.md` | `pm/versions/` 的目录导航、职责边界和结构说明 |
| `PM当前版本计划.md` | 当前 active 版本指针和短状态快照 |
| `PM每日任务清单.md` | 每日只需执行一次的固定动作清单 |
| `daily-execution-history/` | 每日任务执行结果，一天一份 |
| `state/` | 当前经营摘要、控制状态、阻塞和短指针 |
| `expertise/` | 渠道、作家、审稿、托管协作、能力建设等专业资产 |
| `templates/` | 日常表和后续版本治理模板 |
| `versions/<version>/` | 单版本完整计划、产物索引、门禁、runbook、审稿、指标、决策和复盘 |

## 目录治理原则

1. `PM版本推进计划.md` 是最高治理规则，只写跨版本维护规则，不写单个版本现场长内容。
2. `PM当前版本计划.md` 只保留 active 版本指针和短快照，不承载完整版本计划。
3. 一个版本对应一个目录；具体版本内容写入 `pm/versions/<version>/`。
4. 每日任务与每轮主线分离：每日任务一天一次；每轮主线按当前最高价值动作重新判断。
5. `novel/` 是唯一正文事实源；PM 目录不得维护公开候选稿、内部候选包或后台替换候选源的并行正文。
6. 新写、重写、整合、审稿和发布准备都算生产链，但必须分层记录 `new_words / rewrite_words / integrated_words / reviewed_words / publish_ready_words / published_words`。
7. 能由脚本完成的主线草案、字数统计、目录检查和日常表草案优先自动化，PM token 用于质量、剧情、角色和经营判断。
8. `state/` 只放运行态摘要，不放长历史；`expertise/` 只放可复用专业资产，不放单轮现场。

## 推荐读取顺序

### 常规开工

1. `pm/README.md`
2. `pm/PM当前版本计划.md`
3. `node scripts/生成每日主线判断草案.js --root=. --format=markdown`
4. 按脚本推荐主线再定向读取 `pm/state/`、`pm/versions/<active_version>/版本计划.md` 或相关 runbook。

### 触发时读取

1. 治理原则、目录职责或生产计量变化：读 `pm/PM版本推进计划.md`。
2. 切版、历史路径、目录归属不清：读 `pm/PM版本目录导航.md`。
3. 发现低效、空转、顶层污染或真相源重复：读 `pm/PM版本推进注意事项.md`。
4. 每日固定动作窗口：读 `pm/PM每日任务清单.md`。
5. 当前阶段小说写作/审核/运维托管规则：读 `pm/expertise/小说生产托管与协作方法.md`。
6. 渠道、文笔、审稿或能力建设问题：读 `pm/expertise/` 中对应文件。
