# 小说生产托管与协作方法

- 更新时间：`2026-05-05T02:05:00+08:00`
- 适用阶段：`pm_managed_write_review_operate`
- 文档定位：小说编写 / 审核 / 运维的当前阶段协作方法

## 当前阶段结论

从 `2026-05-05` 起，小说编写、审核和普通运营维护先由 PM 直接托管。后续是否恢复多人协作，由用户明确通知后再切换；PM 不自动把 writer/reviewer/operator 派发链恢复为主路径。

## 职责归属

1. PM 直接承担 writer：新写、重写、补写、删改、整合、章节衔接和正文连续性处理。
2. PM 直接承担 reviewer：基础质量审查、商业钩子审查、角色口吻审查、AI 味模板清理、连续性检查和发布前门禁。
3. PM 直接承担 operator：平台发布准备、只读回查、读者信号记录、日常巡检、阻塞证据留存和非敏感内容运营。
4. 用户只处理方向校准、关键经营分歧和敏感动作。

## 真相源规则

1. `novel/` 是唯一正文真相源。
2. 正文不得写进 `pm/` 作为并行候选稿。
3. PM 治理目录只记录计划、指标、门禁、审稿、状态和经营证据。
4. 平台材料必须从 `novel/` 当前正文机械切取，不维护公开候选稿副本。

## 生产流程

1. 先读 `pm/PM当前版本计划.md`、自动化主线草案和命中的当前专题文件。
2. 若主线命中正文生产，直接读取 `novel/我在修仙宗门做成本审计/正文/V1/001-020章.md`。
3. 每轮只选择一个最高价值正文目标，例如第9-15章中段重写、第17章恢复、公开章样板复核或下一顺序章发布准备。
4. 正文改动直接落入 `novel/`，不另建公开候选稿。
5. 改完必须自审：字数、内部材料词、重点 AI 味模板、角色口吻、连续性、替换窗口和发布状态。
6. 自审通过后更新 `pm/state/`、`pm/PM当前版本计划.md`、当前版本 `metrics/` 与 `decisions/`。
7. 每轮写入当日记忆。

## 审核口径

1. PM 自审可以作为短期托管质量证据。
2. PM 自审不能冒充“正式 reviewer 终态”。
3. 若某门禁明确要求 `graph / audit / run` 或外部 reviewer 真证据，PM 自审只能写成 `pm_self_checked`，不得写成 `reviewed_go`。
4. `publish_ready` 只在本地门禁、替换窗口和平台材料条件都满足后才能写。
5. 已发布章替换窗口冻结时，即使正文重写完成，也只能写 `rewrite_integrated_pending_review`。

## 运维口径

1. 普通内容运营由 PM 直接处理，不逐条等待用户确认。
2. 章节发布设置里的 `是否使用AI` 默认选择 `是`。
3. 平台只读回查、字数记录、错别字记录、状态记录和读者信号记录由 PM 直接维护。
4. 平台动作遇到系统风险、敏感声明或法律/财务事项，必须暂停并留阻塞证据。

## 敏感动作边界

以下事项仍由用户本人处理或到场：

1. 密码、验证码、cookie/token。
2. 实名材料。
3. 签约合同。
4. 首发/独家声明。
5. 平台法律声明。
6. 收款、银行卡、提现、税务。

## 协作恢复规则

1. 当前阶段不自动恢复多人协作。
2. 用户明确通知恢复协作模式后，PM 再重新启用 writer/reviewer/operator 分工。
3. 恢复协作后，`novel/` 仍是唯一正文真相源，PM 仍保留兜底直写和自审权。
4. 若恢复后的外部协作连续两轮无正文、无复核、无平台证据或制造多真相源，PM 自动切回托管模式，并记录原因。

1. 本文件是小说协作方法的当前真相源之一，调整托管或协作规则时必须提交到 Git。
2. 只提交与小说协作方法、PM 状态和必要记忆相关的文件，不把无关工作区脏改动混进方法提交。
3. 若 `novel/` 作为独立 Git 仓库存在，正文改动应在 `novel/` 仓库单独提交；根仓只记录 PM 方法、治理状态和必要 gitlink。
