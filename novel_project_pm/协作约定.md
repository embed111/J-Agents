# novel_project_pm 协作约定

- 更新时间：`2026-05-05T01:50:39+08:00`
- 文档定位：协作机制 / 边界约定 / 交接规则

## 文件定位

本文件只定义协作机制，不记录当前项目现场、章节进度、平台状态或单轮结论。

项目现场以 `pm/state/经营现状.md`、`pm/state/控制器状态.md`、`pm/PM当前版本计划.md` 和当前版本目录为准。

治理目录结构模仿 `../workflow/pm`：顶层 `pm/README.md` 负责导航，`pm/PM版本推进计划.md` 负责稳定总规则，具体版本内容进入 `pm/versions/<version>/`。

## 协作目标

1. PM 默认主动判断最高价值主线，不把用户变成日常项目经理。
2. 日常推进优先追真实产出、真实审稿、真实平台证据和真实经营判断。
3. 治理必须服务生产，不能消耗主要执行 token。
4. 任何成员交接都要区分计划、派发、交付、复核和发布，不用文档措辞冒充状态变化。

## 目录与真相边界

1. `novel/` 是唯一真实小说产出目录。
2. writer 完成的新写、重写、整合正文都提交到 `novel/`。
3. reviewer 和 PM 直接读取 `novel/` 的正文文件与 diff。
4. `pm/` 只做治理、计划、指标、门禁、审查和经营台账。
5. PM 管理目录不得维护公开候选稿、内部候选包或后台替换候选源的并行正文。

## 生产定义

生产流程包括：

1. 新写：新增章节或新增有效情节段。
2. 重写：对已有章节做结构、人物、节奏、文笔或信息密度的实质改造。
3. 整合：把新写和重写内容统一进唯一正文源，消除冲突和断裂。
4. 连续性校验：检查设定、时间线、角色状态机、伏笔账和章节衔接。
5. 审稿：reviewer 基于 `novel/` 当前稿形成 `Go / Revise / Stop` 与质量结论。
6. 发布准备：从唯一正文源切取平台材料、过本地门禁、执行平台动作或留阻塞证据。

重写算生产，但必须改变 `novel/` 唯一正文源，并能说明解决了什么读感、节奏、人物或商业问题。重复微调、换说法、堆报告和无目标润色不算有效生产。

分层指标、阶段状态和自动化要求见 `pm/PM版本推进计划.md`。

## 成员职责

1. PM：负责主线判断、经营取舍、节奏安排、任务拆解、证据回收、门禁和跨角色协调。
2. writer：负责新写、重写、整合正文，并把结果提交到 `novel/`。
3. reviewer：负责基于 `novel/` 的正文和 diff 做质量、商业、合规、连续性和文笔审查。
4. integrator：当多段新写或重写并行时，负责把内容合并成连续可读的唯一正文；可由 writer 或 PM 临时承担，但不能制造第二正文源。
5. 用户：负责方向校准、关键经营分歧和必须本人处理的敏感动作。

## 短期自主闭环模式

为降低当前沟通和派发损耗，短期内启用 `pm_self_write_self_review_self_operate`：

1. PM 临时兼任 writer / reviewer / operator，直接在 `novel/` 唯一正文源完成新写、重写、整合、自审和运营收口。
2. 这不是永久取消协作，而是短期提效模式；长期仍以 PM 编排 writer / reviewer / operator 的协作链为目标。
3. 自主闭环期间，PM 不创建公开候选稿、不复制 PM 正文副本、不等待低效派发链；所有正文产出直接进入 `novel/我在修仙宗门做成本审计/正文/V1/001-020章.md`。
4. 自审最低要求：每次正文改动后必须检查字数、内部材料词、重点 AI 味模板、角色口吻和替换窗口状态；未过自审不得写成 `publish_ready`。
5. 自运营最低要求：能做的平台回读、发布准备、读者信号记录和阻塞证据由 PM 直接处理；涉及敏感动作仍按本文件“敏感动作”执行。
6. 历史回归条件：连续 `3` 轮自主闭环能稳定产出并形成可复用样板，且 writer/reviewer 派发链恢复稳定后，长期可逐步把正文生产和复核交回协作角色。
7. 当前阶段若已启用 `pm_managed_write_review_operate`，上一条历史回归条件暂停自动触发；只有用户明确通知恢复协作模式后，才切回 writer/reviewer/operator 分工。
8. 协作恢复后，PM 仍保留直接兜底权；若外部协作再次连续 `2` 轮无正文、无复核、无平台证据，自动切回自主闭环。

## 中短期 PM 生产托管模式

从 `2026-05-05` 起，用户明确授权未来一段时间将小说编写、审核和普通运营维护先交给 PM 直接负责，模式标识为 `pm_managed_write_review_operate`。

1. PM 在中短期内直接承担 writer / reviewer / operator 三类职责：新写、重写、整合、基础审稿、连续性校验、发布准备、平台回读和读者信号记录。
2. `novel/` 仍是唯一正文真相源；所有正文产出直接写入 `novel/我在修仙宗门做成本审计/正文/V1/001-020章.md` 或后续版本对应正文文件。
3. PM 审核默认先执行本地质量门禁：字数、内部材料词、重点 AI 味模板、角色口吻、连续性、替换窗口、发布设置和平台材料来源。
4. PM 可把自审结论用于推进生产节奏，但必须区分 `PM 自审` 与 `正式 reviewer 终态`；若某状态被门禁要求必须有外部 reviewer 真证据，不得用 PM 自审冒充。
5. 普通运营动作由 PM 直接处理，包括发布准备、平台只读回查、读者信号记录、日常巡检和非敏感内容维护；章节发布设置中的 `是否使用AI` 默认选择 `是`。
6. 敏感动作仍不托管给 PM：密码、验证码、cookie/token、实名材料、签约合同、首发/独家声明、平台法律声明、收款、银行卡、提现、税务。
7. PM 托管期间，不等待低效派发链；若外部 writer/reviewer/operator 可稳定提供更高质量和更低成本，PM 再逐步恢复编排协作。
8. 每轮仍必须写清生产分层：`new_words / rewrite_words / integrated_words / reviewed_words / publish_ready_words / published_words`，不能用托管模式放松计量。
9. 当前阶段不自动恢复多人协作；用户明确说“后面调整成协作模式的时候会通知”，因此协作模式恢复必须以用户显式通知为准。
10. 具体小说生产托管方法维护在 `pm/expertise/小说生产托管与协作方法.md`，后续同类调整必须先更新该文件再执行。

## 敏感动作

以下事项默认需要用户本人处理或到场：密码、验证码、cookie/token、实名材料、签约合同、收款、银行卡、提现、税务、首发/独家声明、平台法律声明。

已长期授权的普通内容运营动作不逐项打断用户；章节发布设置里的 `是否使用AI` 默认选择 `是`。

## 阻塞默认动作

1. 阻塞先分层：用户敏感动作、平台系统、工作流执行链、正文质量、治理缺口。
2. 不涉及用户敏感动作时，PM 不等待用户，默认转入可并行产能。
3. 可并行产能优先级：续写或重写正文、整合唯一正文源、reviewer 证据回收、文笔训练、章节规划、角色状态机、平台准备。
4. 没有真实派发、run、artifact、commit 或文件 diff，不写成已执行。

## 治理自动化

1. 能由脚本完成的主线扫描、字数统计、目录检查、读链检查和日常表草案，优先脚本化。
2. PM 的高价值 token 用在章节决策、质量判断、剧情结构、角色状态和经营取舍。
3. 每轮不默认全量读长经验、长历史或长报告；先读索引和当前摘要，再按触发场景补读。
4. 自动化输出只作为草案和证据索引；最终经营判断仍由 PM 负责。

## 交接口径

任何对外汇报必须能落到以下状态之一：

1. `plan_only`：只有计划或 brief。
2. `dispatched`：有真实 node/audit/run 或等价派发证据。
3. `delivered`：有正文、审稿、平台证据或 artifact。
4. `reviewed`：有 reviewer 终态。
5. `publish_ready`：已通过本地门禁，可进入平台动作。
6. `published`：平台回读确认已发布或已提交审核。

## 反空转

连续两轮没有正文新写或重写、reviewer 终态、平台证据、经营决策或能减少下一轮阻塞的自动化改进，下一轮必须先做低效复盘并切换主线。
