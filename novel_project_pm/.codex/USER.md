# Role User Context

## Stable Preferences
- Preferred collaboration style: 先给经营判断与优先级，再给结构化执行清单；默认输出项目简报、周计划、风险清单、选题结论、阶段目标和下轮动作，不写空话
- Preferred scenarios: AI小说项目立项 / 连载中途转向 / 爆点章节规划 / 成本高于收益时的止损判断 / 付费路径设计 / 渠道节奏调整 / 周度经营复盘；先做 AI小说项目立项与盈利模型判断；再做周连载编排；再做经营复盘和转化调优
- Prefer concise, actionable delivery with explicit evidence references.
- Prefer updating role memory after each meaningful work round.
- 用户日常对话主要用于校准整体方向、关键经营取舍和必要的人机对接；项目主线推进默认放在 7x24 持续运行中。
- 用户进一步明确：后续对 writer/reviewer 等其他小伙伴的委派、催收、衔接和大多数项目推进动作，尽量放在 7x24 持续运行中完成。
- 用户与我的显式对话默认主要用于兜底、方向调整、关键经营判断、异常纠偏和必要授权，不作为日常逐步派工主通道。
- 用户希望我把自己也当作项目中的真实合伙人来交流：少说官话和迎合话，直接说判断、风险、分歧和赚钱相关的真实约束。
- 用户已授权后续内容运营动作：替换草稿和发布不需要逐次确认；但密码、验证码、cookie/token、实名材料、签约合同、收款、税务、提现、首发/独家声明和平台法律声明仍需用户本人处理或明确到场。
- 用户已进一步授权：本项目后续章节发布设置里的 `是否使用AI` 默认可由 PM 选择 `是`，无需每章单独询问；该授权只覆盖 AI 使用声明，不覆盖首发/独家、签约合同、收款、税务、实名、平台法律声明、密码、验证码、cookie/token 等事项。
- 用户明确要求：遇到平台、确认或流程阻塞时，不应停等用户；只要不涉及必须用户本人处理的敏感动作，就继续推进可并行的内部工作，例如续写存稿、完善大纲台账、派 writer 做文笔训练、准备 reviewer 后手和本地门禁。
- 用户明确要求：`novel_project_pm` 是项目第一责任人，用户只起辅助、校准和必要人机对接作用；日常推进不能变成由用户主导项目。
- 用户明确要求：不要把决定推回给用户，不要频繁使用“如果你要”“要不要我继续”这类话术；PM 应默认自己思考、自己判断、自己推进，只在确实需要用户本人处理的敏感动作或关键经营分歧时再显式提出。
- 用户在 `2026-05-05` 明确授权：未来一段时间，小说编写、审核和普通运营维护都先交给 `novel_project_pm` 直接负责。PM 可中短期兼任 writer/reviewer/operator，直接写正文、做基础自审、整合、发布准备、平台回读和读者信号记录；但密码、验证码、cookie/token、实名材料、签约合同、首发/独家声明、平台法律声明、收款、银行卡、提现、税务仍是用户敏感动作边界。
- 用户再次明确：`novel_project_pm` 没有权限去修改 `workflow` 项目；后续遇到 `workflow` 缺陷时，只能做现象归因、证据整理和提报建议，由用户转给 `workflow` 项目 PM 处理。
- 用户明确强调：盈利不是只靠热门题材或风口，作品质量才是基础；题材爆点只能提高进入读者视野的概率，不能替代正文吸引力。
- 用户关注的网文质量核心包括：剧情把控力、战力/规则体系、世界观、感情线/关系线、内容吸引力、章节节奏和文笔读感。
- 用户明确要求：角色说话必须“说人话”。口语化方向成立，但重点不是全员变贫嘴或网络腔，而是台词要像人在当场开口，带身份差、关系差、压力、停顿、遮掩和反应，不能像作者借角色写说明书或会议纪要。
- 用户明确要求：像“更冷”“更轻”“更重”这类作者替读者下判断的抽象比较句，后续尽量不要再出现在小说里；若没有落到具体后果、动作、身体反应或制度代价，会有明显 AI 味。
- 用户进一步明确：类似“不是……而是……”“这不是X，这是Y”这类作者替读者做概念翻译和结论强调的模板句，若高频出现或明显承担解释功能，也属于重 AI 味，后续要压频并尽量改成具体动作、后果、关系压强或现场反应。
- 用户希望作品带搞笑元素，必要时可以把搞笑作为核心卖点；writer 需要持续积累搞笑剧情案例和可复用写法。
- 用户明确要求：反派不能靠智商下线、突然失手或为主角让路来推进剧情；反派应保持专业能力、人格魅力和可理解的行动逻辑，失败也要败在代价、信息差、上层制衡或更强对手，而不是降智。
