# 需求详情-统一入口与训练优化模块

## 1. 主题目标与价值说明
1. 主题目标：
   1. 在 `workflow` 中建立“统一入口”，先展示全量 agent 信息，再进入训练优化流程。
   2. 将“agent 展示/版本治理”和“训练闭环执行”放在同一业务域内闭环，但保持子模块边界清晰。
   3. 首期支持手动训练计划触发，后续平滑并行接入分析模块自动生成计划。
2. 业务价值：
   1. 训练动作建立在可见的 agent 资产事实（版本、能力）之上，降低盲训风险。
   2. 训练任务来源可追溯（手动/自动），执行顺序可控（优先级队列）。
   3. 人工可干预（标注相似、移除任务），减少自动化误触发带来的治理成本。

## 2. 用户画像与使用场景
1. 用户画像 A：训练负责人
   1. 关注点：快速识别应训练的 agent、创建计划并推进执行。
2. 用户画像 B：训练师（`../trainer`）
   1. 关注点：按队列接收任务、执行训练、回写结果。
3. 用户画像 C：平台治理者
   1. 关注点：agent 版本变更与训练结果可追溯、可审计。
4. 核心场景：
   1. 打开统一入口后先看全部 agent 概览角色卡（头像式矢量图 + 文本：当前版本、核心功能），视觉上接近“名片/工牌”。
   2. 点击某个 agent 查看完整发布史与功能迭代变更。
   3. 手动填写训练计划并入队。
   4. 训练师按优先级队列执行训练，人工可移除无效任务。
   5. 后续分析模块自动生成计划，与手动计划并行入队。

## 3. 用户旅程或关键流程
1. 用户进入 `训练中心` 统一入口，系统加载全量 agent 概览。
2. 用户选择 agent，查看详情（版本历史、功能变更）。
3. 用户点击“训练优化”，进入 `训练闭环` 模块（非会话流程）。
4. 用户手动创建训练计划（首期主路径），提交入队并填写优先级。
5. 系统按优先级调度训练队列（同优先级按入队时间顺序）。
6. 训练师匹配 `../trainer`（首期字符串匹配），执行训练并回写。
7. 若检测到相似计划，仅标注提示，默认仍允许执行。
8. 用户可人工移除队列任务（无二次确认，需风险提示和审计留痕）。
9. 后续分析模块可自动生成计划，并与手动计划并行入队。

## 4. 功能需求清单（编号）
1. FR-UO-01 统一入口与子模块分层
   1. 一级入口为 `训练中心`。
   2. 入口内至少包含两个子模块：
      1. `Agent 资产与版本`
      2. `训练闭环`
2. FR-UO-02 Agent 概览角色卡展示
   1. 必须展示根目录下全部 agent。
   2. 每个 agent 必须以“角色卡”形式展示，至少包含：
      1. 头像式矢量图标识（人物头像语义，非抽象几何图标）
      2. `agent_name`
      3. `current_version`
      4. `core_capabilities`
      5. `last_release_at`
   3. 角色卡布局应采用“名片/工牌”风格，强调“每个 agent 是真实工作角色”的识别感。
3. FR-UO-03 Agent 详情与完整发布史
   1. 点击 agent 后必须可查看完整版本发布史。
   2. 每个版本至少展示：版本标识、发布时间、功能迭代变更说明。
4. FR-UO-04 Agent 工作区发布版本治理（首期）
   1. 版本控制对象是 agent 工作区。
   2. 版本区域只展示“已发布迭代”（发布版本时间线 + 迭代变更说明）。
   3. 视图中不展示中间 Git 提交记录（commit 明细、逐条提交列表）。
   4. 非 Git 工作区需明确标记 `git_unavailable`，不阻断训练入口。
5. FR-UO-05 训练流程与会话流程解耦
   1. 训练流程独立于会话流程。
   2. 训练对象是 agent 工作区能力，不绑定单一会话上下文。
6. FR-UO-06 手动训练计划触发（首期主路径）
   1. 用户可手动创建训练计划并触发入队。
   2. 手动计划最小必填字段：
      1. `target_agent`
      2. `capability_goal`
      3. `training_tasks`
      4. `acceptance_criteria`
      5. `priority`
7. FR-UO-07 训练师指定与匹配
   1. 训练师来源目录固定为 `../trainer`（首期）。
   2. 首期匹配方式采用字符串匹配，并展示匹配结果列表。
8. FR-UO-08 优先级训练队列
   1. 所有任务入队必须填写优先级（必填）。
   2. 优先级分级固定为 4 级：`P0/P1/P2/P3`。
   3. 调度规则：先按优先级，再按入队时间。
9. FR-UO-09 手动与自动计划并行入队
   1. 队列任务来源至少支持：`manual`、`auto_analysis`。
   2. 两类来源并行存在，统一进入训练队列调度。
10. FR-UO-10 相似计划标注机制（弱去重）
   1. 系统应检测并标注相似计划。
   2. 相似计划默认不强制去重、不自动拒绝。
11. FR-UO-11 训练任务人工移除
   1. 队列内所有任务支持人工移除。
   2. 移除动作不要求二次确认弹窗。
   3. 必须给出风险提示，并记录审计日志（操作者、时间、任务、原因可选）。
12. FR-UO-12 执行回写与审计
   1. 每次训练执行必须回写：`status/result_summary/run_ref`。
   2. 关键链路必须可追溯：`agent -> version -> plan -> queue_task -> run_result`。
13. FR-UO-13 入口门禁
   1. 未选定 agent 时，不允许创建训练计划。
   2. trainer 无匹配结果时，可保存计划但不可启动执行。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 性能：
   1. Agent 概览首屏加载 `p95 <= 3s`（100 个 agent 基线）。
   2. 队列入队写回结果 `p95 <= 2s`。
2. 可靠性：
   1. 队列调度必须幂等，重复提交不得生成重复 `queue_task_id`。
   2. 训练执行失败允许重试，失败历史需保留。
3. 可观测性：
   1. 所有队列动作（入队、移除、开始、完成）需事件留痕。
   2. 相似计划标注规则和命中结果可查询。
4. 安全与治理：
   1. Git 数据读取仅限配置的 agent 工作区根路径。
   2. 移除任务虽无二次确认，仍需强审计留痕，避免不可追责操作。

## 6. 验收标准（Given/When/Then）
1. AC-UO-01 统一入口首屏
   1. Given 用户打开 `训练中心`
   2. When 首屏渲染完成
   3. Then 可看到全部 agent 角色卡（头像式矢量图 + 版本 + 核心功能），且整体为名片/工牌风格
2. AC-UO-02 Agent 详情版本史
   1. Given 用户点击任一 agent
   2. When 进入详情页
   3. Then 可查看完整发布史和功能迭代变更
3. AC-UO-03 已发布迭代视图
   1. Given agent 工作区存在 Git 仓库
   2. When 查看版本区域
   3. Then 仅可查看已发布迭代时间线与对应变更摘要，不显示中间 commit 明细
4. AC-UO-04 非 Git 兼容
   1. Given agent 工作区无 Git 仓库
   2. When 查看版本区域
   3. Then 显示 `git_unavailable`，且不阻断训练入口
5. AC-UO-05 手动计划入队
   1. Given 用户填写手动计划最小字段
   2. When 点击入队
   3. Then 成功创建队列任务并记录任务来源为 `manual`
6. AC-UO-06 优先级必填与四级
   1. Given 用户尝试入队但未填写优先级
   2. When 提交
   3. Then 系统拒绝并提示“优先级必填”
   4. And 仅允许 `P0/P1/P2/P3`
7. AC-UO-07 调度顺序
   1. Given 队列中存在多个任务
   2. When 调度器执行
   3. Then 按优先级和入队时间顺序处理
8. AC-UO-08 相似计划标注
   1. Given 新计划与已有任务相似
   2. When 入队后查看任务
   3. Then 任务被标注“相似计划”，但默认仍可执行
9. AC-UO-09 人工移除
   1. Given 队列中任意任务
   2. When 用户点击移除
   3. Then 不弹二次确认，显示风险提示并移除任务，写入审计日志
10. AC-UO-10 手动与自动并行
   1. Given 同时存在 `manual` 与 `auto_analysis` 来源任务
   2. When 查看训练队列
   3. Then 两类任务均可见并参与统一调度
11. AC-UO-11 训练流程与会话解耦
   1. Given 某 agent 无活跃会话
   2. When 创建训练计划并执行
   3. Then 训练流程可独立运行且对象为 agent 工作区能力
12. AC-UO-12 验收证据完备性
   1. Given 任一 AC 完成并进入验收回传
   2. When 提交验收证据
   3. Then 每条 AC 必须至少包含：界面截图路径、接口请求/响应示例、关键日志或数据库记录路径、对应代码文件路径
   4. And 关键交互链路（至少 `AC-UO-01/02/05/09`）必须附录屏或等效 GIF 证据
   5. And “门禁统计表/汇总看板截图”仅可作为辅助证据，不可替代该 AC 对应的功能页面截图
   6. And 验收通过并完成复核后，可执行证据精简，仅保留门禁截图与汇总文件

## 7. 边界条件与异常处理
1. 根目录扫描不到任何 agent：
   1. 显示空态与修复提示（检查 `agent_search_root`）。
2. 版本历史过长：
   1. 默认分页加载，避免详情页阻塞。
3. trainer 匹配结果为空：
   1. 允许保存计划，但禁止启动执行。
4. 训练任务被人工移除后：
   1. 不可自动恢复；如需恢复需重新入队。
5. 相似计划误判：
   1. 仅作为提示，不影响执行；后续通过规则迭代优化。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 根目录下 agent 可被稳定发现。
   2. agent 工作区可访问发布迭代元数据，且“已发布迭代”判定固定为 `tag + release note`。
   3. `../trainer` 目录结构可被读取用于首期匹配。
2. 开放问题：
   1. `core_capabilities` 首期来源是否仅依赖 `AGENTS.md` 自动提取。
      1. 推断/假设（置信度: 中）：首期建议“自动提取 + 人工修正”并存。
   2. 自动生成计划（`auto_analysis`）是否默认要求人工审核后执行。
3. 能力扩展评估（skill/agent）：
   1. 新增 skill：`training-queue-governance-gate`
      1. 触发理由：优先级必填、相似标注、人工移除等规则需长期一致校验。
      2. 采用成本：中。
      3. 采用决策：`采用`（待后续实现阶段创建）。
   2. 新增 agent：专职“队列调度审计 agent”
      1. 触发理由：后续自动任务规模扩大时，审计复杂度提升。
      2. 采用成本：中到高。
      3. 采用决策：`暂不采用`（Phase 0 保持轻量）。

## 9. 本轮增量需求（2026-03-11）- 训练闭环页与客户端式单窗口展示
1. 问题定义（用户明示）：
   1. 当前 `训练闭环` 页若继续按“训练师匹配 + 队列执行”展示，无法承载“参考资料 / Skill 工作集 / 多用例三轮评分 / 下一轮或回退”的闭环训练流程。
   2. 页面展示应保持与现有 `训练中心` 一致的“单窗口客户端视图”，而不是拆成多个互相独立的大页面容器。
   3. `参考资料与 Skill 工作集` 属于长内容区，必须使用区内滚动条，而不是把整页无限拉长。
2. 本轮收敛口径（用户明示 + 推断/假设，置信度: 高）：
   1. `训练闭环` 的主模型由“入队执行”升级为“闭环迭代”。
   2. `trainer` 不再是训练主流程门禁对象，降级为可选协作角色或责任人。
   3. 训练通过判定固定为：清空上下文执行 3 轮后，`avg_score >= threshold`。
   4. 若 `avg_score` 高于上一轮但仍未达阈值，则进入下一轮。
   5. 若 `avg_score` 低于上一轮，则回退本轮新增的参考资料与 Skill，不回退整个工作区。
   6. 页面主视图必须在“单个浏览器窗口”内同时展示训练闭环核心区块；长内容以区内滚动承载。
   7. 为降低信息密度，`训练闭环` 内应区分 `创建任务` 与 `任务状态` 两种工作模式，但仍保持在同一主窗口内切换。
3. 新增功能需求：
   1. FR-UO-14 训练闭环对象层
      1. `训练闭环` 必须显式提供以下对象：`训练计划`、`参考资料与 Skill 工作集`、`训练用例`、`评分与决策`、`迭代记录`。
      2. 以上对象允许按工作模式分层呈现，而不是全部堆叠在同一可视层级。
      3. 不得继续仅以 `trainer 匹配 + 队列任务 + 执行结果` 作为页面主结构。
   2. FR-UO-15 Codex 资料搜索与入库判断
      1. 系统必须支持由 `Codex` 对资料候选进行搜索、过滤与入库价值判断。
      2. 资料来源至少支持：`URL`、本地文件、`GitHub`、可搜索路径。
      3. 页面中必须可区分：`候选资料`、`已采纳资料`、`本轮新增资料`。
   3. FR-UO-16 三轮清空上下文评分机制
      1. 每轮训练的评分必须由 3 次“清空上下文后的独立执行”组成。
      2. 页面必须展示 `Run1 / Run2 / Run3 / Avg`。
      3. 单轮最高分不得直接代替本轮训练结果。
   4. FR-UO-17 下一轮与回退决策
      1. 页面必须展示本轮平均分、上一轮平均分、阈值、分差与决策结果。
      2. 当平均分下降时，必须支持“回退本轮新增参考资料与 Skill”。
      3. 回退范围必须显式展示为“仅本轮增量”。
   5. FR-UO-18 单窗口客户端式展示
      1. `训练闭环` 页面必须保持与现有 `训练中心` 一致的“单主窗口”视觉结构。
      2. `训练中心` 标题必须位于主窗口内部左上区域，展示口径与 `角色资产与版本` 页一致。
      3. 页面不得依赖整页向下长滚动来承载核心训练内容。
      4. 当前工作模式对应的核心区块必须在桌面单窗口视图内同时可见。
   6. FR-UO-19 区内滚动规则
      1. `参考资料与 Skill 工作集` 必须为独立滚动区。
      2. 当训练用例或迭代记录超出可见高度时，应优先使用区内滚动，而不是继续拉高页面总高度。
   7. FR-UO-20 创建任务与任务状态分区
      1. `训练闭环` 在同一主窗口内必须提供 `创建任务` 与 `任务状态` 两种工作模式。
      2. `创建任务` 负责 `计划名称`、`目标能力`、`通过阈值`、`初始用例集`、`初始参考资料 / Skill` 与 `启动首轮`。
      3. `任务状态` 负责 `当前任务摘要`、`参考资料与 Skill 工作集`、`三轮评测`、`评分决策` 与 `迭代记录`。
      4. 存在活跃训练任务时，默认进入 `任务状态`；新建任务通过模式切换进入 `创建任务`。
   8. FR-UO-21 训练任务列表与切换
      1. `训练闭环` 必须提供可见的训练任务列表，用于查看其他训练任务。
      2. 任务列表至少展示：`任务名称`、`目标能力`、`当前状态`、`最近轮次 / 最近分数`。
      3. 用户必须可从列表中切换当前查看的训练任务，不要求通过返回上一级页面完成。
      4. `创建任务` 入口必须与任务列表相邻，保证用户能明确区分“看已有任务”和“新建任务”。
   9. FR-UO-22 任务状态页的阅读顺序
      1. `任务状态` 默认应先展示 `当前任务总览` 与 `当前判定`，再进入工作集、评测结果和迭代记录等详细内容。
      2. 详细内容应按二级标签或等效分区承载，避免所有内容在一个视图中同时展开。
      3. 用户进入状态页后，应能在 3 秒内判断“当前看的是哪个任务、当前结论是什么、下一步动作是什么”。
4. 新增验收标准：
   1. AC-UO-13 训练闭环主结构
      1. Given 用户进入 `训练中心 -> 训练闭环 -> 任务状态`
      2. When 页面渲染完成
      3. Then 可同时看到 `当前任务摘要`、`参考资料与 Skill 工作集`、`训练用例与评分`、`迭代记录`
   2. AC-UO-14 工作集区内滚动
      1. Given 候选资料数量超出工作集区可见高度
      2. When 用户查看 `参考资料与 Skill 工作集`
      3. Then 资料列表在该区块内部出现滚动条
      4. And 不应把整页继续向下拉长作为主要承载方式
   3. AC-UO-15 三轮平均分判定
      1. Given 用户执行一轮训练评测
      2. When 系统返回结果
      3. Then 页面展示 `Run1 / Run2 / Run3 / Avg`
      4. And 以 `Avg` 作为当前轮次判定依据
   4. AC-UO-16 回退本轮新增项
      1. Given 当前轮平均分低于上一轮

      3. Then 用户可直接识别当前训练流程的阶段顺序与当前所在阶段
      4. And `创建任务` 与 `任务状态` 使用一致的阶段语义与步骤条风格
      5. And 当前阶段状态区可直接回答“现在到哪一步、这一步在做什么、接下来做什么”
      6. And 低优先历史明细仍通过折叠或区内滚动承载，不挤占流程主阅读线

## 17. 本轮增量需求（2026-03-11）- 训练流程需展示分叉、回退与收敛路径
1. 问题定义（用户明示）：
   1. 仅用线性阶段流仍不足以表达训练中的真实演进。
   2. 用户明确要求看到类似 `git` 图的效果，展示训练如何发散、哪些分支被回退、哪些分支继续进入下一轮并最终收敛。
2. 本轮收敛口径（用户明示 + 推断/假设，置信度: 高）：
   1. 训练流程不仅有线性阶段，还存在“分支探索 -> 劣化回退 -> 保留有效分支 -> 继续下一轮 -> 最终收敛”的路径演进。
   2. `任务状态` 页首屏应能回答：
      1. 当前轮之前尝试过哪些训练分支；
      2. 哪些分支已被回退；
      3. 当前保留的是哪条主线；
      4. 为什么能进入下一轮或为什么仍未收敛。
   3. 该图的目标是表达决策路径与训练演化，不是仅作为装饰性时间线。
3. 新增功能需求：
   1. FR-UO-30 训练演进图
      1. `任务状态` 页应提供类似 `git` 图的训练演进图，用于展示训练分叉、回退和收敛路径。
      2. 训练演进图至少应区分以下节点或等效语义：
         1. 基线/上一轮起点；
         2. 分支探索；
         3. 劣化分支；
         4. 回退节点；
         5. 保留继续的主线；
         6. 当前所在轮次/判定节点；
         7. 未来收敛目标。
      3. 对于“训练提升但评分不足”的场景，图中必须能看出该主线为何继续进入下一轮，而不是被误读为已完成收敛。
      4. 对于“训练后能力劣化”的场景，图中必须能看出该分支已被回退，以及回退影响的是哪一轮新增工作集或哪一批训练增量。
      5. 图中各节点应附带最少量但足够判断的摘要信息，例如：
         1. 轮次；
         2. 关键得分；
         3. 决策动作（回退/继续/完成）；
         4. 简短原因。
      6. 演进图应与上方阶段流保持语义一致：阶段流回答“现在在闭环的哪一步”，演进图回答“是沿着哪条训练路径走到这一步”。
4. 新增验收标准：
   1. AC-UO-27 训练演进路径可见
      1. Given 用户进入 `训练中心 -> 训练闭环 -> 任务状态`
      2. When 页面展示某个已进入评测或迭代中的训练任务
      3. Then 用户可直接识别该任务的训练分叉、已回退分支与当前保留主线
      4. And 对于“劣化后回退”的分支，用户能看出该分支已被放弃
      5. And 对于“提升但不足以通过”的主线，用户能看出其继续进入下一轮而非已收敛完成
      6. And 演进图与当前阶段状态区共同支持用户判断“为什么走到这里、接下来怎么走”

## 18. 本轮增量需求（2026-03-11）- 右侧栏收口为当前演进节点详情
1. 问题定义（用户明示 + 已确认推进方向）：
   1. 仅展示演进图还不够，用户还需要快速看到“当前选中节点”的具体判定与动作。
2. 本轮收敛口径（用户明示 + 推断/假设，置信度: 高）：
   1. 右侧栏不应再只是独立于流程图的泛化结果卡。
   2. 右侧栏应默认服务于当前选中演进节点，帮助用户从图上的“点”直接读到该节点的结论、原因、动作和影响范围。
3. 新增功能需求：
   1. FR-UO-31 演进节点详情联动

         1. 节点名称；
         2. 节点所属轮次；
         3. 当前得分或关键指标；
         4. 判定结论；

      3. Then 右侧栏直接展示该节点的详细判定、下一步动作与关键原因
      4. And 用户无需再从多个分散卡片拼凑当前节点结论
      5. And 右侧信息与演进图中的当前主线/回退关系保持一致
