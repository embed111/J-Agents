# 需求概述 - 飞书文档优化 Agent

## 1. 项目背景与问题定义
1. 背景：
   1. 用户希望新增一个专用 agent，帮助优化本人可访问的飞书文档。
   2. 当前飞书开放平台 API 链路对用户来说不可用或难以落地，因此不能把第一阶段实现建立在 API 接入成功之上。
2. 当前问题：
   1. 私有飞书文档无法由当前会话直接读取，导致“文档优化”无法直接在聊天里完成。
   2. 用户需要的是“先能用”的辅助优化能力，而不是先完成一套标准平台集成。
   3. 若没有结构化 agent，单篇文档优化会依赖人工复制正文，流程摩擦较高。

## 2. 目标用户与核心场景
1. 目标用户：
   1. 企业内部员工，能够访问本人权限范围内的飞书 Wiki / Doc 文档。
   2. 希望在本机环境中，用已登录飞书会话辅助优化文档。
2. 核心场景：
   1. 输入一篇飞书文档链接，agent 读取正文与结构，输出优化建议。
   2. 在用户确认后，agent 将优化稿写回原文档。
   3. 对单篇文档执行标题、摘要、层级、措辞、信息密度优化，而不是接管整个知识库。

## 3. 业务目标与成功指标
1. 业务目标：
   1. 降低“把飞书正文搬出来再优化”的操作摩擦。
   2. 让单篇文档优化形成“读取 -> 分析 -> 草稿 -> 确认 -> 写回”的稳定闭环。
   3. 在不依赖开放平台 API 的前提下，先落地可用版本。
2. 成功指标（第一阶段）：
   1. 用户输入单篇飞书链接后，agent 能成功读取正文并生成结构化优化建议。
   2. 用户确认后，agent 能把优化稿写回同一篇文档。
   3. 全流程在用户本机执行，不要求输入飞书账号密码到 agent。
   4. 每次写回前都有人工确认，不出现“静默覆盖原文”。

## 4. 范围界定
1. In Scope：
   1. 单篇飞书 Wiki / Doc 文档优化。
   2. 浏览器自动化读取正文与结构。
   3. 输出优化建议与改写稿。
   4. 人工确认后写回原文档。
   5. 本机已登录会话复用。
2. Out of Scope：
   1. 飞书开放平台 API 集成作为第一阶段主链路。
   2. 批量扫描整个知识库。
   3. 自动归档、自动目录治理、跨空间治理。
   4. 无确认自动写回。
   5. 保存或托管飞书账号密码。

## 5. 优先级与阶段规划
1. Phase 1：单篇文档只读优化
   1. 输入链接。
   2. 读取正文。
   3. 输出问题诊断与优化建议。
2. Phase 2：单篇文档确认后写回
   1. 生成优化稿。
   2. 用户确认。
   3. 写回同一文档并留痕。
3. Phase 3：增强能力（暂不进入本轮）
   1. 模板化优化。
   2. 跨文档一致性检查。
   3. 多文档批处理。

## 6. 风险、依赖与待确认事项
1. 依赖：
   1. 用户本机可运行浏览器自动化工具。
   2. 用户本机已有可用的飞书登录态。
   3. 目标文档在用户权限范围内。
2. 风险：
   1. 飞书页面 DOM 变化可能导致正文抽取失效。
   2. 登录态过期、验证码或二次验证可能中断流程。
   3. 自动写回若缺少显式确认，存在误覆盖风险。
3. 待确认事项：
   1. 第一版支持的浏览器范围是否限定为 Chrome / Edge。
   2. 写回策略是“整篇替换”还是“块级定点替换”。
   3. 是否需要保留原文快照或差异预览。

## 7. 能力扩展评估
1. 新增 agent 角色：`采用`
   1. 角色名称建议：`飞书文档优化 Agent`。
   2. 采用理由：该能力与现有需求分析师、workflow UI 主题不同，职责清晰且可独立演进。
2. 新增 skill：`暂不采用`
   1. 第一阶段先用浏览器自动化闭环验证能力边界。
   2. 若后续出现重复的 DOM 提取、写回安全治理，再拆 skill。
