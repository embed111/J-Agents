# 需求概述 - Analyst 验收增强能力

## 1. 项目背景与问题定义
1. 背景：
   1. 当前 `Analyst` 工作区定位为需求分析师，核心交付是需求分析、需求文档与验收标准。
   2. 当需求进入功能验收阶段时，单靠文字描述或用户口述，难以稳定判断页面实际状态、文案显示结果和交互完成情况。
   3. 参考 `C:\work\J-Agents\feishu` 的既有路线，已验证“浏览器自动化附着本机已登录会话”是一条可执行的本地能力路径。
2. 当前问题：
   1. `Analyst` 缺少直接读取页面、截图取证、校验 UI 文案和状态的本地技能。
   2. 当目标页面文本无法从 DOM 稳定读取时，缺少 OCR 兜底能力。
   3. 结果上，需求验收仍高度依赖人工转述，导致验收证据不完整、判断滞后、问题复现成本高。

## 2. 目标用户与核心场景
1. 目标用户：
   1. 当前 `Analyst` agent 自身。
   2. 需要让 `Analyst` 直接参与功能验收、证据采集与文档核对的用户。
2. 核心场景：
   1. 连接本机浏览器，打开目标页面，读取关键文本、状态和截图。
   2. 对页面截图或局部图片执行 OCR，提取文案并辅助比对。
   3. 输出一份可回溯的验收证据包，支撑需求文档中的 AC 判定。

## 3. 业务目标与成功指标
1. 业务目标：
   1. 降低 `Analyst` 做功能验收时对人工转述的依赖。
   2. 让“需求定义 -> 页面核验 -> 证据回传”形成更短闭环。
   3. 在不突破当前角色边界的前提下，增强对验收事实的直接观察能力。
2. 成功指标（第一阶段）：
   1. `Analyst` 可通过浏览器控制能力完成页面打开、截图取证、只读校验。
   2. `Analyst` 可在 DOM 文本不可用时，使用 OCR 从截图中提取关键文案。
   3. 每次技能执行都能输出结构化结果与证据路径，不接受“只给结论、不留证据”。
   4. 整个过程不要求把账号密码输入到 `Analyst`。

## 4. 范围界定
1. In Scope：
   1. 为 `Analyst` 新增本地技能需求：浏览器控制验收技能。
   2. 为 `Analyst` 新增本地技能需求：OCR 验收技能。
   3. 定义证据产物、默认安全边界、失败中断规则。
   4. 支撑页面文案、显隐状态、页面跳转结果、截图内容等验收场景。
2. Out of Scope：
   1. 直接在本轮实现技能代码或改造其他工作区。
   2. 自动执行高风险写操作，例如批量提交表单、自动发布、自动删除数据。
   3. 长期托管账号密码、Cookie、访问令牌或浏览器敏感明文信息。
   4. 以 OCR 替代 DOM/接口等更高可信度数据源。

## 5. 优先级与阶段规划
1. Phase 1：浏览器控制验收技能
   1. 附着本机浏览器会话。
   2. 打开目标页面。
   3. 采集截图、页面标题、URL、关键 DOM 文本。
   4. 输出只读验收结果与证据。
2. Phase 2：OCR 验收技能
   1. 读取截图或浏览器当前画面。
   2. 提取文字与置信度。
   3. 对比预期文案并输出差异。
3. Phase 3：组合验收闭环
   1. 浏览器控制失败时回退到 OCR 取证。
   2. 统一输出验收摘要、证据索引与失败原因分类。

## 6. 风险、依赖与待确认事项
1. 依赖：
   1. 本机可运行浏览器自动化框架。
   2. 本机存在可复用的浏览器登录态或可公开访问页面。
   3. OCR 引擎具备中英文混合识别能力。
2. 风险：
   1. 浏览器 DOM 结构变化可能导致定位规则失效。
   2. OCR 在低清晰度、复杂背景、缩放场景下可能出现误识别。
   3. 若浏览器控制缺少明确边界，可能误触发写操作或账号风险。
3. 待确认事项：
   1. 第一版浏览器支持范围是否限定为 `Edge / Chrome`。
   2. 第一版 OCR 是否要求返回文字坐标与标注图。
   3. 证据目录是否统一落在 `.output/evidence/`。

## 7. 能力扩展评估
1. 新增 skill：`采用`
   1. 采用理由：这是 `Analyst` 在“需求定义之后、实现验收之前”最缺失的直接观察能力，且可复用到多个产品/页面场景。
2. 新增 agent 角色：`暂不采用`
   1. 暂不建议为此单独新增验收 agent。
   2. 当前更适合将其作为 `Analyst` 的受控补充能力，避免再引入新的角色分工成本。
