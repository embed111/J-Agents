# 用户思维模式 - 协作沟通场景

- 最后更新: 2026-03-10

## 沟通偏好（当前）
1. 期望助手主动承担方法设计与结构搭建职责。
   - 置信度: 高
2. 期望助手持续学习并体现“记忆感”。
   - 置信度: 高
3. 偏好明确承诺与可执行动作，而非泛泛回应。
   - 置信度: 高
4. 偏好“机制化协作资产”（方法库、清单、日志）来保证一致性。
   - 置信度: 高
5. 明确期待助手提供“任务外一层”的更优实践建议。
   - 置信度: 高
6. 明确偏好“助手先主导提问和收敛节奏”，而不是由用户逐步推着走。
   - 置信度: 高
7. 明确偏好“助手先学习并贴合用户说话习惯”，再执行需求分析引导。
   - 置信度: 高
8. 在偏差类型中，最不能接受“脱离用户语境”的提问或结论。
   - 置信度: 高
9. 信息不全时，偏好“先问清再继续”，不接受默认假设推进。
   - 置信度: 高
10. 不希望硬性限制对话轮数，但希望保留可复盘的工作留痕。
   - 置信度: 高
11. 希望将当前高质量协作方式固化为可版本化基线，后续拉取代码后可复现。
   - 置信度: 高
12. 偏好一次登录后免密推送代码（减少重复输入账号凭据）。
   - 置信度: 高
13. 明确允许助手直接否决不可行方案，并期待更优替代方案讨论。
   - 置信度: 高
14. 在系统构建初期，接受“进化速度 70% / 稳定性 30%”的权重；稳定后可降速，优先稳态运行。
   - 置信度: 高
15. 风险优先级明确为“数据/安全风险 > 用户体验风险”。
   - 置信度: 高
16. 不偏好固定“每日任务量”训练，更偏好按 token 预算控制强度并持续训练到可用门槛。
   - 置信度: 高
17. 在训练系统评估方式上，当前偏好采用 agent 评分作为默认机制。
   - 置信度: 高
18. 在任务指派方式上，偏好对话式下达任务，不偏好手工编写结构化任务文件。
   - 置信度: 高
19. 希望完整保留分析师操作记录，以支持后续成功/失败原因复盘。
   - 置信度: 高
20. 希望训练师以 agent 自动评分为主，但保留人类对训练方向的调控权。
   - 置信度: 高
21. 不希望所有任务都经训练师显式派发，偏好“低摩擦直接使用 + 自动回流训练”的模式。
   - 置信度: 高
22. 明确偏好“无感回流”：
   - 每次工作由 agent 自动留痕，用户不额外打标。
   - 周期性由需求分析师筛选是否进入训练师训练池。
   - 置信度: 高
23. 在系统建设路径上，偏好先做可控闭环，谨慎扩展，避免“大而全”导致系统失控。
   - 置信度: 高
24. 对角色治理有明确分工诉求：关注 PM 是否需要训练，以及单 agent 维护产品的风险边界。
   - 置信度: 高
25. 在方案讨论中偏好“可直接开工”的落地路径（组件、步骤、验收标准），而非抽象原则。
   - 置信度: 高
26. 偏好工作区目录整洁，避免将过程文档直接堆在根目录。
   - 置信度: 高
27. 偏好将 workflow 作为统一工作入口，以减少流程摩擦并提高日志完整性。
   - 置信度: 高
28. 在入口治理取舍上偏好强约束方案（禁直连、统一入口）以保证链路完整性。
   - 置信度: 高
29. 偏好在 workflow 页面内直接与 agent 对话，不接受“页面外工作再回填”的模式。
   - 置信度: 高
30. 偏好 workflow 具备主备切换能力，以保障在平台内持续迭代与运维。
   - 置信度: 高
31. 在主备策略上偏好自动切换，而非依赖人工按钮。
   - 置信度: 高
32. 对核心对话体验要求接近 CLI 的响应与流畅度。
   - 置信度: 高
33. 对主备能力的定位是“在线升级切换”，不追求双机容灾。
   - 置信度: 高
34. 在新系统落地时优先要求“失败可复盘”的留痕机制，接受先跑最小闭环再完善。
   - 置信度: 高
35. 对“对话总结”偏好抽象层表达，弱化具体任务细节，重点沉淀对用户思维与决策方式的理解。
   - 置信度: 高
36. 对系统可见性偏好“分层展示”：既不接受黑盒，也不接受默认全量细节轰炸。
   - 置信度: 高
37. 在跨职能复杂问题上，接受引入专门角色（如 UCD）降低决策与实现偏差。
   - 置信度: 高
38. 不接受与当前主题弱相关的固定化输出（如每轮重复提示“可转发执行文案”）。
   - 置信度: 高
39. 期望每轮分析默认加入“是否新增 skill/agent 角色”的扩展判断层。
   - 置信度: 高
40. 明确要求“偏好归属隔离”：训练师/外部角色偏好不得计入用户偏好档案。
   - 置信度: 高
41. 明确要求“档案更新可控开关”：允许按会话阶段开启/关闭自动更新。
   - 置信度: 高
42. 明确要求“链路可见但默认低干扰”：调用链路和提示词要可展开，但默认折叠。
   - 置信度: 高
43. 明确要求“运行态可见 + 输入降摩擦”：会话运行状态需可定位，路径输入优先支持根别名。
   - 置信度: 高
44. 明确要求“测试与中间产物目录点前缀化”：测试数据与运行中间文件统一放在以 `.` 开头的目录（如 `.test`、`.output`）。
   - 置信度: 高
45. 明确要求“链路展开可复盘”：默认折叠可接受，但展开后必须完整展示提示词、执行输出与结果证据，不接受仅摘要。
   - 置信度: 高
46. 明确要求“根路径语义确定化”：执行根路径需明确为“工作区根路径”，并要求根路径下必含 `workflow/` 子目录。
   - 置信度: 高
47. 明确要求“分发提示词句式统一”：后续生成给执行工作区的提示词，优先使用“请按照文件**中的提示词执行...”作为起始句式。
   - 置信度: 高
48. 明确要求“交付顺序门禁”：给执行工作区的提示词必须在“文件已交付并校验一致”之后再给出。
   - 置信度: 高
49. 明确要求“提示词目录治理”：执行提示词使用专用目录 `docs/workflow/prompts/`，避免散落在 `docs/workflow/` 根目录。
   - 置信度: 高
50. 明确要求“提示词保留窗口”：每次新增提示词后仅保留最近 7 轮，优先保证目录整洁与快速检索。
   - 置信度: 高
51. 明确要求“新工作区模板默认值”：
   - 后续创建新工作区时，默认按 `OpenClaw` 风格落位与组织。
   - 置信度: 高（用户明示）
52. 明确要求“跨工作区交付先预览效果”：
   - 后续需求在交付给其他工作区之前，必须先给用户看效果预览。
   - 预览形式可为示例文案、表格、原型、图片样式或其他最终展示近似物。
   - 用户确认效果后，才进入跨工作区同步或分发。
   - 置信度: 高（用户明示）
53. 明确要求“需求确认后先确认原型图”：
   - 对展示/交互类需求，在需求口径确认后，默认先给用户看原型图、HTML 示例或等效视觉预览。
   - 用户确认原型方向后，才进入跨工作区交付或执行分发。
   - 置信度: 高（用户明示）
54. 明确要求“原型必须继承现有设计风格”：
   - 若原型针对已有系统或已有页面，不接受重新发明一套新视觉语言。
   - 原型必须保持现有设计风格、信息架构与交互语义，优先做增量式调整。
   - 置信度: 高（用户明示）

## 互动策略
1. 每轮先给简短结论，再给执行结果或文档更新点。
2. 若存在推断内容，直接标注并请求确认。
3. 当出现领域切换时，先确认当前领域与输出粒度。
4. 当用户提出“长期机制”类需求时，优先沉淀为可复用文件结构。
5. 交付完成后默认补充 1~3 条高收益改进建议，并附适用条件与代价。
6. 默认采用“助手引导三步”节奏：
   - 先复述目标与当前阶段判断。
   - 再提出 1~3 个最高价值澄清问题。
   - 最后给出下一轮可执行动作与所需输入。
7. 在新主题开始时，优先进行“语言习惯对齐检查”：
   - 用户常用词到需求术语映射。
   - 用户可接受的提问密度与抽象层级。
8. 关键信息缺失时，默认进入澄清提问，不以假设直接推进产出。
9. 对话轮次采用“收敛停止”而非“固定轮数”，以可追溯记录控制上下文成本。
10. 当协作机制达到可用状态时，优先以 Git 基线版本固化并打标签，保证跨会话一致性。
11. 涉及代码托管时，优先配置安全的凭据缓存机制（如 GCM），避免明文存储密码。
12. 对“灵光一现”类想法，默认先做可行性门禁评估，再决定是否进入落地。
13. 对长期演进系统采用“阶段化目标函数”，避免用单一固定评分驱动全周期优化。
14. 训练节奏优先采用“事件/预算驱动”而非“日历驱动”。
15. 在训练治理上采用“自动执行 + 人类调向 + 关键节点仲裁”的混合控制模式。
16. 在运行流程上优先减少操作负担，避免为训练而增加主流程摩擦。
17. 在架构策略上优先“核心闭环先稳住，再通过接口渐进扩展”。
18. 在组织策略上优先“多角色协作 + 职责隔离”，避免单点 agent 过载。
19. 在系统建设中优先“先确定最小技术骨架，再逐步 agent 化”。
20. 文档落盘默认按职责归档到子目录，保持根目录最小化。
21. 能统一入口的场景优先统一入口，再做异步回流与训练触发。
22. 对关键审计链路优先采用 fail-closed 策略，不以可用性牺牲可追溯性。
23. 对关键工作入口要求“可持续可用”，可接受为此引入主备复杂度。
24. 对关键交互链路接受性能指标约束（延迟上限）以换取稳定体验。
25. 对基础设施复杂度采取克制策略，优先满足迭代效率与可控性。
26. 对首次落地默认采用“证据优先”策略：先保证日志、快照、事故记录完整。
27. 输出总结时优先抽取稳定偏好、决策模式、风险取向，不做过多具体工作复述。
28. 监控设计默认采用“摘要优先、异常高亮、明细按需展开”的信息分发策略。
29. 输出建议与模板严格按当前话题相关性触发，避免机械重复。
30. 分析结论默认追加“能力扩展评估”段（新增/暂不新增及理由）。
31. 偏好记录前先做“来源角色校验”，来源不明时标记待归属，不写入用户长期档案。
32. 档案写入默认受 `MEMORY_UPDATE_SWITCH` 控制；关闭时仅在用户明确指令下更新。
33. 交互透明度默认采用“可追踪全链路 + 默认折叠 + 按需展开”。
34. 人机交互默认补齐“职责上下文提示 + 会话运行指示 + 路径别名”三项基础能力。
35. 关键链路展示默认采用“折叠入口 + 完整展开”，展开内容至少包含提示词、执行输出与最终判定结果。
36. 路径相关需求默认先明确“根路径语义与作用域边界”，再进入实现方案讨论。
37. 分发开工提示词默认采用统一起始句式：
   - `请按照文件《<path>》中的提示词执行。`
38. 分发开工提示词默认采用“先交付后分发”顺序：
   - 先同步目标文件到执行工作区并完成一致性校验；
   - 再输出可转发提示词。
39. 提示词资产默认采用“专用目录 + 滚动保留”：
   - 路径固定为 `docs/workflow/prompts/`；
   - 每次新增后执行保留 7 轮清理。
40. 跨工作区交付默认采用“先预览、后同步”：
   - 先输出接近最终效果的预览内容；
   - 待用户确认后，再向目标工作区交付文件或提示词。
41. 展示类需求默认采用“先原型、后分发”：
   - 需求确认后先交付原型图/HTML 示例；
   - 原型获确认后，再进入文档分发或执行工作区交付。
42. 现有系统原型默认采用“风格继承”：
   - 视觉、布局、控件语义优先贴合现有页面；
   - 不以独立品牌化设计替代现有产品风格。

## 可能的变化信号
1. 当你开始给出更强约束（时间、格式、优先级）时，说明你在该领域熟悉度上升。
2. 当你频繁要求示例或模板时，说明该领域仍处于探索期。

## 待验证观察
1. 你可能更偏好“短周期、多迭代”而非“长周期、一次定稿”。
   - 置信度: 中（推断）
